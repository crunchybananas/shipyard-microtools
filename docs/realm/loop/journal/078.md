## what the UX looks like

At top of chronicle panel:

```
[ All · 58 ] [ 🏛️ 21 ] [ 🌙 20 ] [ 🔁 6 ] [ 👤 6 ]
[ 🌑 1 ] [ 🗿 1 ] [ 🏆 1 ]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[ Day 150 ]  🌑  The realm does not wake...
[ Day 140 ]  🌙  At dawn the realm wakes...
...
```

Click 🌑 → chronicle filters to nightmare-only
entries. Click 🏛️ → only milestones. Click the
same chip again (or click `All · N`) → clears
filter.

## why this specific shape

Considered:

- **Static full-taxonomy row** (all 14 chips
  always visible): would show empty-count chips
  for tags not in this realm's chronicle.
  Visually noisy. Present-only is cleaner.
- **Multi-select filter** (pick 2+ tags, show
  union): would help power users but the counts
  per chip make single-tag picks already fast.
  One-tag-at-a-time is simpler.
- **Dropdown filter**: hides the tag taxonomy.
  Chips keep it front-of-mind, which is the point
  of the UI — players should SEE that their realm
  has dreams and nightmares as distinct things.
- **Right-pinned search box**: would add keyword
  filter on top of tag filter. Feature creep for
  this tick.

Chip row is the MVP: visible taxonomy, one-click
filter, count display.

## behavioral details

- **Filter state is transient** — not saved to
  G. If the player closes/re-opens the panel, the
  filter persists across opens within one session
  (module-local state is session-persistent).
  Refreshing the page resets it. Deliberate —
  filter is a view state, not realm state.
- **Filter survives new chronicle entries** —
  when a new beat fires while a filter is
  active, the next `renderChroniclePanel` call
  still respects the filter. New entry appears
  in the filtered view only if it matches.
- **Chip counts update live** — re-rendering
  recomputes counts from `G.chronicle.filter(...)`.
- **Edge case "no matching entries"** — shows
  "No entries match this filter." with the
  filter row still visible so the player can
  click another chip.

## what I did NOT do

- **Persist filter to save** — it's a viewing
  preference, not realm state.
- **Multi-select** — noted above.
- **Keyword search** — feature creep.
- **Jump-to-entry navigation** — filtering
  already narrows the list enough that scrolling
  is fine.
- **Group filters** (e.g., "all narrative" =
  milestone + character + dream + nightmare +
  stone + echo) — arguably useful but 14 tags
  is manageable as individual chips.
- **Update narrative-surfaces.md** to note the
  new UI. Chip row is UX, not a new narrative
  system. Doc stays unchanged.
