## target

`js/story.js:checkStoryBeats()`. After 022 (added 5
beats) and 025 (added undo-aware story flags), the
function had accumulated 17 nearly-identical 4-line
blocks. Each beat:

```js
if (!hasFlag('firstX') && G.buildings.some(b => b.type === 'X')) {
  setFlag('firstX');
  chronicle('...', 'milestone');
}
```

Copy-pasted 17 times, varying only in flag name, type
string, and chronicle text.

## the shape of the change

Extract the 13 simple first-build beats into a data
table at module scope:

```js
const BUILDING_FIRST_BEATS = [
  { type: 'house',      flag: 'firstHouse',      text: '...' },
  // ... 12 more
];
```

Replace 13 × (4 lines) with 1 × 5-line loop:

```js
for (const beat of BUILDING_FIRST_BEATS) {
  if (!hasFlag(beat.flag) && G.buildings.some(b => b.type === beat.type)) {
    setFlag(beat.flag);
    chronicle(beat.text, 'milestone');
  }
}
```

## what stayed inline and why

Three categories did NOT collapse into the table:

1. **Side-effect beats** (tavern, barracks, church). Each
   fires a `chronicle()` AND an `ensure*()` character-
   setup function. Folding them into the table would
   require adding an optional `after: fn` field — mixed
   data and function references, fragile. They're
   distinct narrative moments; keeping them inline makes
   "this beat also introduces a character" visible at a
   glance.

2. **Special-shape beats** (firstBirth, castleBuilt,
   firstRaidSurvived). These key on `G.stats.citizensBorn`
   or `G.stats.raidsSurvived`, not
   `G.buildings.some(b => b.type === X)`. Different
   predicate. castleBuilt also uses `G.namedCharacters`
   to insert the mayor's name into the chronicle text
   ("Nerys Greyhelm proclaims...") and fires with the
   `'victory'` tag, not `'milestone'`. Folding these
   would require generalizing the beat schema into a
   full mini-interpreter. Not worth it.

3. **Population thresholds** (pop10/25/50/75/100). These
   were ALREADY data-driven pre-026. Didn't touch them.

## why shape matters (the refactor-only discipline)

Before 026, adding a new chronicle beat required:
- Writing 4 lines of repeated if-block at the right
  insertion point in checkStoryBeats
- Wrapping at ~110 chars to match existing style
- Risking a typo in the flag name (`firstBackery`
  breaks the gate silently)

After 026:
- Add one row to `BUILDING_FIRST_BEATS`: `{ type,
  flag, text }`
- That's it. Three fields, one line, type-aligned.

The "N beats we narrate" fact is now a table a human
can read top-to-bottom in 8 seconds. Before, it was
buried in a ~100-line function with side-effect and
special cases interleaved. The refactor makes the
*interleaving* itself visible: the 3 side-effect beats
stand out now, which was the point.

Net LoC: **-33** (+35 added, -68 removed). More shrink
than expected because the repeated `chronicle('...',
'milestone')` pattern collapses cleanly when extracted.

## what I did NOT do

- Refactor the population-thresholds loop (already
  data-driven).
- Generalize to a full story-beat interpreter that
  could handle the 3 special-shape beats. The 3
  specials are legitimately different; forcing them
  into one interface would trade clarity for
  uniformity.
- Touch the `ensure*()` character functions. They're
  separate concerns (who is Nerys Greyhelm? — lives in
  the same file but isn't beat-firing code).
- Add new beats. Refactor-only means no behavior
  change.
