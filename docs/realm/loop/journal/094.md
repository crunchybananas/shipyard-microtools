### 2. Draw code reads the table

```js
const cap = _WINTER_CAPS[b.type] || _WINTER_CAPS._default;
const snowY = s.y + cap.y;
const snowW = cap.w;
ctx.beginPath();
ctx.ellipse(s.x, snowY, snowW, cap.h, 0, 0, Math.PI * 2);
ctx.fill();
// ...icicles logic unchanged
```

Before 094 the residential cluster (house/tavern/bakery/lumber/
market/windmill/granary) all shared the same fallback geometry
(w=14, h=3, y=-28). After 094 each has distinct dimensions on
three axes — width, height, vertical offset.

## why this shape

- **Per-type variation, not per-type draw-path.** Introducing
  different SHAPES (rect for market awning, dome for bakery)
  would require bigger draw-code changes. Ellipse-dimension
  variation achieves the audit-visible disambiguation without
  rewriting the draw logic. 086 filed the variation idea;
  the minimum-viable shape is 3-axis dimensional.
- **Wide axes give maximum silhouette contrast.** Width ranges
  from 18 (windmill) to 34 px (market) — 16px spread. MinY
  ranges from 66 (windmill, tall body) to 75 (market, awning)
  — 9px spread. Each residential has a unique combination
  across these axes.
- **Tower/church retain pre-094 geometry** (w=8, w=10
  respectively) — they were already distinct outliers.
  Table includes them for single-source-of-truth, not
  behavior change.
- **Fallback pattern via `_default`.** Buildings not in the
  table still render (drawGeneric fallback types) using the
  baseline (w=14, h=3, y=-28). Future buildings can opt-in
  by adding a table entry without touching the draw code.
- **Non-destructive for the 051 tavern flag** (which projects
  ABOVE the roofline at y≈-40). Tavern cap at y=-27 stays
  below the flag. Keeps both disambiguation signals stacked
  ("flag on tavern" + "widest residential cap" in winter).
