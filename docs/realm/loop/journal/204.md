### finding 2 (MEDIUM): Same invariant claims multiplicative-only pattern

Same invariant block. The pattern claim:

> Pattern: `const <char>Mult = G.namedCharacters?.<char> ? 1.N : 1;`
> applied inside existing multiplier chains.

**Problem:** 201 bard ships an ADDITIVE variant:
```js
const bardBonus = G.namedCharacters?.bard ? 5 : 0;
G.happiness = Math.min(100, Math.max(10, 50 + bardBonus + ...));
```

This is a flat baseline addition, not a
multiplicative mult-chain. The invariant
claims a single pattern; the code has two.

**203's mitigation:** added an "Additive-baseline
vs multiplicative-bonus" entry to the new
`## observed patterns` section. So the additive
variant IS documented — but in a separate section
that distinguishes between observation (1 use)
and invariant (prescriptive rule).

**Tension:** the invariants section now claims
"the pattern is `<char>Mult`", while observed-
patterns section says "additive is a sibling
variant filed at 1 use." These are inconsistent.
Reader sees: which is it?

**Severity MEDIUM** because:
- 203 doc edit was fresh; can be repaired in
  same archivist sweep as finding 1.
- A reader running `<char>Mult` boilerplate for
  a hypothetical NEW mechanic might miss the
  additive option.
- Not load-bearing for current code, but
  load-bearing for future authoring.

**Recommended fix:** adjust the invariant's
pattern claim to acknowledge both shapes:
> Pattern (multiplicative variant): `const
> <char>Mult = G.namedCharacters?.<char> ? 1.N : 1;`
> applied inside existing multiplier chains.
> Pattern (additive baseline variant; see
> observed patterns): `const <char>Bonus =
> G.namedCharacters?.<char> ? N : 0;` added to
> a baseline expression. Currently 1 use
> (bard); promote to invariant at 3+.
