# 064 — the approach (surprise, pre-event dream bias)

**Challenge:** Surprise-shaped. Un-filed idea
proposed under the Evolution mandate — last surprise
was 059 (4 ticks ago, at threshold). Not in
ideas.md; the concept emerged from noticing that
039 dreams, 043 nightmare, 044 time-aware, and 056
stone were all separate systems that had no
temporal interaction. If the nightmare fires on day
150, the day-140 dream has zero knowledge that
something's coming.

064 wires them together: in the 10-day approach
window before either once-per-realm beat, the dream
pool gets a thematic nudge. Warning threads for
the nightmare. Founding + hearth threads for the
stone. Players paying attention will *feel* the
approach in the chronicle's tone shift before the
beat itself arrives. Players who aren't paying
attention get a normal dream.

No chronicle announcement. No tutorial hint. Pure
ambient build-up.

## the build

`docs/realm/js/story.js` — new `_approachingEventBoost()`
function + one 3-line integration in `checkDreamBeat`.

```js
export function _approachingEventBoost() {
  const kname = G.kingdomName || 'Realm';
  const boost = [];
  if (!hasFlag('nightmare_fired')) {
    const nDay = _dreamHash(`${kname}_nightmare_day`) % 200 + 50;
    if (G.day >= nDay - 10 && G.day < nDay) boost.push('warning');
  }
  if (!hasFlag('stone_found')) {
    const sDay = _dreamHash(`${kname}_stone_day`) % 170 + 30;
    if (G.day >= sDay - 10 && G.day < sDay) {
      boost.push('founding');
      boost.push('hearth');
    }
  }
  return boost;
}
```

Called alongside 044's `_realWorldDreamLens` boost
inside `checkDreamBeat`:

```js
// 044 seasonal boost
for (const name of lens.boost) {
  if (avail.includes(name)) avail.push(name);
}
// 064 approach boost
for (const name of _approachingEventBoost()) {
  if (avail.includes(name)) avail.push(name);
}
```

Same boost mechanism 044 uses — duplicate the
thread name in `avail[]`, doubling its selection
probability per hash-slot pick.

## the window is one dream cycle

Initial design used a 5-day window, which caught
zero dream-fires because 055's cadence=10 means
dreams only fire on multiples of 10. For nightmare
day 150, the [145, 149] window contained no
dream-firing days.

Widened to **10 days** = exactly one cadence cycle
so the LAST dream before the event lands in the
window. For nightmare day 150, day 140's dream
gets the boost. For stone day 176, day 170's
dream gets the boost. One boosted dream per
lifetime event — deliberate; a wider window would
make the boost overbearing.

### aggregate content signal

16 kingdoms tested. Each kingdom's seeded last-
before-event dream day fired, and content checked
for warning/founding/hearth regex matches.

```
                  N    warn    founding  hearth
control (day 60)  16   50%     38%       44%
nightmare appr.   16   56%     —         —
stone approach    16   —       31%       56%
```

Approach lifts (vs control):
- Nightmare → warning: **+6pp**
- Stone → hearth: **+12pp** ← clearest signal
- Stone → founding: −7pp (within noise; founding
  is gated-always so it gets lots of control
  picks)

The 12pp hearth lift is the design target — clear
enough for an attentive player to notice, quiet
enough that missed.

### why the lift is modest

Doubling a thread's weight in a pool of 8+ available
threads gives a ~12pp single-slot pick bump. With
3 image slots and image dedup, the net effect on
"dream contains this thread's imagery" is roughly
+10pp — exactly what we measure.

A bigger bias (triple the weight, or exclusive
approach-only threads) would read as the game
*telling* the player something's coming. The
modest lift is the difference between "the realm
feels off today" and "warning: nightmare
incoming." 064 picks the former.

## what I did NOT do

- **Add a chronicle hint or toast.** Would make
  the approach loud and break the attentive-
  player-only philosophy.
- **Make the boost larger than ×2.** Tripling
  would telegraph the event. The whole point of
  a seeded nightmare is the reveal on the event
  day.
- **Apply the approach to the cycling dreams
  themselves** (i.e., boost each dream-day's own
  thread choices to match "last few days before
  the next nightmare cycle"). The nightmare and
  stone are once-per-realm, not cyclical.
- **Extend the window beyond 10 days.** Would
  catch multiple dreams in the approach. One
  pre-echo is enough; two feels overdone.
- **Add approach detection for 059's echoes.**
  Interesting idea (echo would surface
  related-thematic prior beats when nightmare is
  approaching), but would add complexity. Filed
  below.
