## the strategic question

The user steered 2026-04-28: "in some loops can
we really focus on the svg generation. opus 4.7
has expanded capabilities so we should rethink
them or think about the 3d engine version."

Three possible interpretations:

1. **"Focus on SVG/3D as ENRICHED EXPLORATION"**
   — sandbox is the deliverable; never integrate.
   Goal is to demonstrate what's possible, not to
   ship live.
2. **"Focus on SVG/3D toward LIVE INTEGRATION"**
   — sandbox is a stepping stone; sprites
   eventually replace canvas drawX in the live
   game.
3. **"Focus on SVG/3D as PARALLEL TRACK"** —
   sprites accumulate; integration happens IF and
   WHEN it makes sense, judged per-building.

The user hasn't clarified. The loop has been
operating on interpretation #2 implicitly (with
"sandbox-first" as a durable invariant from 161).

## evaluating the three interpretations

### #1 (enriched exploration only)

**Pro:** lowest risk; clear stopping point (when
all 11 buildings have sprites); clean separation
of layers.
**Con:** the SVG investment never reaches players;
arguably expensive. ~1059 lines of SVG sitting in
sandbox with no user-facing value is hard to
defend if a player can't ever see them.
**Verdict:** insufficient given Opus 4.7's
expanded capabilities — the user wouldn't have
flagged "rethink them" if exploration alone were
the goal. Reject.

### #2 (toward live integration)

**Pro:** SVG investment justifies itself; players
see richer art; the cross-axis triangle pattern
becomes load-bearing; the loop has a concrete
finish line.
**Con:** integration is HARD. 165's render-
layers.md identified 6 unsolved concerns:
day/night tint, winter caps (158's
dome-conforming arc!), hover halo, fog modulation,
per-instance variation (kingdom-hash glass palette
on church; kingdom-hash bell on church top), and
animation (canvas pennants/flag/blades sway).
**Verdict:** TARGET, but not this tick.
Integration is a multi-tick sprint that should
land WHEN we have a critical mass of sprites.
Currently 5/11; 6-8 sprites is the sweet spot.

### #3 (parallel track)

**Pro:** flexible; per-building decisions; doesn't
commit to wholesale replacement.
**Con:** no clear endpoint; risk of permanent
sandbox-limbo; partial integration creates two
visual styles in the live game (some buildings
SVG, others canvas).
**Verdict:** valid as a TRANSITION strategy
during integration sprint, NOT as long-term
strategy. Use briefly.

## recommendation

**Adopt interpretation #2 (toward live
integration), executed in three phases:**

### Phase A: Complete the sprite roster (~5 ticks)

Continue current cadence to ship sprites for the
remaining 6 buildings (house, tavern, blacksmith,
market, bakery, barracks; possibly farm). At 1
sprite per ~2-3 ticks with variety in between,
this phase ends ~tick 180-185.

This builds critical mass before integration. A
half-roster integration (5 SVG buildings + 6
canvas buildings simultaneously rendered) creates
visual inconsistency that's worse than either
end-state.

Continue 3D mesh work in parallel where it adds
value (windmill blade animation; sun rotation;
the .glb reconciliation question — file but
defer).

### Phase B: Integration sprint (~3-5 ticks)

Once 9-11 sprites exist, ship a series of
integration ticks. Order:

1. **Integration scaffolding tick** — render.js
   gains a `_USE_SVG_SPRITES` feature flag (default
   off) and a sprite-loader path. ~30 LoC.
   No visible change.
2. **Day/night tint composition** — SVG sprites
   composite under the existing canvas multiply-
   blend tint. Test on granary alone.
3. **Winter cap composition** — 158's dome arc
   applies on top of SVG granary in winter.
   Tests the layer-composition approach.
4. **Hover halo + fog** — final composition
   concerns.
5. **Per-instance variation** — church glass
   palette as CSS custom property; kingdom-hash
   sets the property at building-level. Or
   separate variant SVG files.
6. **Live-game enable** — flip `_USE_SVG_SPRITES`
   to true; all 11 buildings ship as SVG.

Each step has a rollback (set flag back to
false). Risk-bounded.

### Phase C: Animation + polish (~2-3 ticks)

After integration, ship:
- Animated windmill sails (167-filed; needs
  shader work in canvas pipeline OR `<animate>`
  in SVG)
- Castle pennants + flag sway
- Tower lantern flicker + banner sway
- Final polish review with chrome-mcp

### What this means for ticks 171-180

- **172-180 (~9 ticks)**: ship 6 more SVG sprites

  interleaved at 30-40% rate.
- **~tick 180**: integration sprint begins.
- **~tick 186**: integration complete; live game

If user redirects mid-arc, this plan adapts. The

## the 3D engine question (separate axis)

The user said "or 3D engine version." 3D has
received less direct work (2 ticks: 163, 167) but
has 7/11 building meshes. Strategic question:

- Is the 3D prototype a future canvas
  REPLACEMENT (eventually realm runs in WebGL2
  with the full 11-building roster), OR
- A separate exploration that stays standalone?

**Recommendation: standalone.** 3D and 2D canvas
are different games architecturally — switching
would mean rewriting input, citizens, render
order, UI. The 3D prototype is valuable as
exploration, not as a migration target.

The .glb meshes that change outside the loop
should be **explicitly noted in
render-layers.md as "loop does not touch"** — a
boundary marker that prevents future ticks from
trying to reconcile them.

## what I am NOT recommending

- **Halt SVG axis now.** 5/11 is too few for
  integration; aborting at 5 wastes the
  investment.
- **Replace canvas drawX before integration is
  proven.** The 6-step sprint above tests
  composition before commitment.
- **Animate sprites before integration.**
  Animation only matters if SVG ships live.
  Defer until phase C.
- **Pursue 3D-as-replacement.** Architecture cost
  exceeds visual gain.
- **Force a tick-180 deadline.** The dates above
  are pace estimates; user steering could
  accelerate or decelerate.
