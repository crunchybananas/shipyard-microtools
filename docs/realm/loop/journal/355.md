## findings

### MEDIUM-actionable

**[play] harness can't reliably
probe long-day beats: random
events (drought/plague) kill realm
mid-probe.**

Probe-1 setup: defense=999 +
nextRaidDay=9999 + resources=99999.
Ran fastForward(240). Final state:
- day=241 (target reached)
- year3 fired
- **realm_fell fired** (population
  hit 0)
- droughtChron + plagueChron events
  triggered earlier in run
- **target_348_fired = false**
  (gate: year3 + day>=160)
- **target_352_fired = false**
  (gate: autumn + year2 + day>=50)
- **target_354_fired = false**
  (gate: year2 + day>=80)

Root cause: `checkStoryBeats()`
guard at story.js:40 returns early
when `G.realmEnded === true`. With
random events still active under
raid-suppression setup, the realm
can starve from drought + plague
before reaching late-day gates.

**Follow-up control** (pop pinned
to 50 each 50-day segment) confirms
all 3 beats fire correctly:
```
{
  target_352_fired: true,
  target_354_fired: true,
  target_348_fired: true,
  pop: 50, restorations: 1
}
```

So the beat-firing logic itself is
correct (verify-logic 78/78 also
confirms). The issue is **probe-
environment fragility**: long-
running [play] probes can't
reliably reach late-day state
without an event-suppression knob.

**Filed 355 [code]**: add
`G.debug.disableEvents = true`
knob analogous to nextRaidDay
=9999. ~5-10 LoC in events.js
+ debug surface. Closes the
analog gap to 311 everHadBuilding
(which closed building-gate
fragility but not population-
vitality fragility).
