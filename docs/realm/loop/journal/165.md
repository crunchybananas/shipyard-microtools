## doc structure

10 sections:

1. **Status / why this exists** — when written,
   user-steering provenance, memory pointer
2. **The three layers** — canvas (live, 30
   buildings), SVG (sandbox post-161, 3
   sprites), 3D WebGL2 (parallel prototype, 6
   building meshes). Per-layer: entry point,
   form, strengths, weaknesses, currently
   shipped count.
3. **`assets/meshes/`** — the .glb files exist
   but the 3D prototype generates its own
   procedural meshes; reconciliation is a
   future-tick concern.
4. **The cross-axis triangle** — granary in all
   three layers (158 canvas / 161 SVG / 163 3D)
   establishes the precedent. Pattern: when one
   layer gets meaningful work, pair the work in
   another within ~5 ticks.
5. **Coverage map** — 14-row table of canvas /
   SVG / 3D per building. Calls out gaps:
   8 of 11 game buildings still need SVG; 5 of
   11 still need 3D meshes; barn-vs-farm naming
   discrepancy between 3D and canvas.
6. **Authoring-cost data** — concrete LoC table
   for granary / castle / church across canvas
   and SVG. SVG is ~43% more lines for ~2× more
   detail per building. 3D mesh is ~15 LoC per
   building at much lower fidelity.
7. **When to pick each layer** — decision rules:
   canvas = default for live-ship work; SVG =
   when the form benefits from gradients /
   patterns / elliptical-arc paths; 3D = for
   spatial/lighting/atmospheric exploration.
8. **Integration concerns NOT yet solved** —
   day/night tint, winter caps, hover halo, fog
   modulation, per-instance variation,
   animation. Each is named so a future
   integration tick has a checklist.
9. **Invariants** — cross-axis pairing, sandbox
   before integration, 3D prototype stays
   standalone, detail-per-zoom holds for SVG by
   construction.
10. **Related loops** — chronology 016 → 165
    pulling together the threads.

## why this shape

- **Mirror narrative-surfaces.md / audio-
  surfaces.md.** Both have status/sources/
  invariants/related-loops sections. New doc
  follows the same template so future ticks
  navigating loop docs find consistent
  structure.
- **Coverage map is the single most actionable
  section.** Future ticks deciding "what next
  for the SVG axis?" can read the map and pick
  an empty cell. Same for 3D.
- **Authoring-cost table grounds the SVG/canvas
  comparison in DATA.** ~445 canvas LoC vs ~638
  SVG LoC for the same 3 buildings — the
  numbers say SVG is ~43% more lines but with
  meaningfully more detail per line. Future
  ticks can defend SVG choices with this row.
- **Integration-concern checklist names the
  unsolved problems.** When a future
  integration tick comes, it has a list to work
  through, not a blank page.
- **Cross-axis triangle invariant** is the
  emergent rule from 158/161/163 — codified
  here so future ticks don't re-derive it.
- **Memory + doc paired.** Memory file
  `project_realm_loop_render_axis.md` captures
  USER STEERING; this doc captures TECHNICAL
  STATE. Both needed; both maintained
  separately.
