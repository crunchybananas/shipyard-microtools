### LOW (1) — rebuild-mutex gap: original beats can double-fire after fallback

**Discovered by code-audit.**

319 fallback beats (sea_bell_lost +
church_step_worn_lost) include a UNI-
DIRECTIONAL mutex against the originals:

```js
// 319-A: gates on !storyFlags.sea_bell_known
trigger: G => year3 && day>=62 && everHadBuilding.church
  && !buildings.some(church) && !sea_bell_known
```

The originals (sea_bell_known + church_
step_worn_known) gate ONLY on year3 +
buildings.some(church) — they DO NOT check
the corresponding *_lost flag.

**Scenario where this matters:**
1. Player builds church.
2. Realm reaches d=62 BEFORE building any
   year3 + church configuration; church is
   destroyed by raid.
3. Realm reaches y3 (d=57) and church-absent
   d=62 → sea_bell_lost fires.
4. Player rebuilds church post-y3.
5. Next checkStoryBeats: year3=true,
   church present → sea_bell_known
   gate satisfied → ALSO fires.

Result: chronicle has BOTH "There is a
bell at the church..." AND "The realm
remembers a bell that was pulled from
the sea. The church that held it is
gone." — narrative inconsistency.

**Severity LOW** — rebuild-after-loss is
an uncommon scenario; chronicle is
read-once; the inconsistency is mild.
But the design intent (2-beat MUTEX) is
violated unidirectionally.

**Mitigation (~4 LoC)**: extend the
originals' gates to check the *_lost
flag:

```js
// sea_bell_known
trigger: G => year3 && buildings.some(church)
  && !storyFlags.sea_bell_lost_known

// church_step_worn_known
trigger: G => year3 && buildings.some(church)
  && !storyFlags.church_step_worn_lost_known
```

Bi-directional mutex; both directions
of "first-fired wins" preserved. Filed
as 323 [code] for next fixer.

**Verify-logic addition needed**: a
test asserting original beats DO NOT
fire when corresponding *_lost flag
is set. Pattern matches 319's mutex
test #55 but flipped.
