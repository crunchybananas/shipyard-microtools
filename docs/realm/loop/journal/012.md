# 012 — the-fixer (tint hue-variation)

**Challenge:** the-fixer — accepted: close 011's own
recommendation. 011 (the-contrarian) argued the luminance
curve should stay put and the tint overlay should hue-vary
instead. This tick implements that — ~10 lines in
render.js, graphics-axis, no mechanic changes.

## what

`render.js:2515-2530` — the night-overlay multiply block.
Before: three static coefficients `(160, 150, 100)` giving
a uniform blue-cast at every non-plateau moment of the day.

After: the coefficients come from a dayT-dependent lookup:

- **Dawn** (`t < 0.10`): soft rose → `kR=80, kG=140, kB=160`
  (R drops least, carries warmth forward as the sun rises)
- **Dusk → night lerp** (`t >= 0.10`): interpolate from
  amber (`80, 120, 180`) at t=0.70 to indigo (`180, 140,
  100`) at t=1.0. `nightBlend` clamps to [0, 1].

Magnitude (`cappedDarkness`) is unchanged — the overlay
is not darker or lighter; only the color shifts. The 0.7
darkness cap stays in force (playability-load-bearing per
011).

## verification (chrome-mcp, summer scene, darkness≈0.30)

Numeric sampling of the new `(tintR, tintG, tintB)` at
representative dayT points:

| dayT | tint                | feel          |
| ---- | ------------------- | ------------- |
| 0.02 | rgb(231, 213, 207)  | warm rose     |
| 0.10 | rgb(231, 219, 201)  | warm amber    |
| 0.70 | rgb(231, 219, 201)  | warm amber    |
| 0.75 | rgb(226, 218, 205)  | transitioning |
| 0.85 | rgb(216, 216, 213)  | near neutral  |
| 0.99 | rgb(202, 213, 224)  | cool indigo   |

Prior behavior at darkness=0.30 was a constant
`rgb(207, 210, 225)` (cool blue) across *every* tinted
moment. The new output swings from warm to cool across the
day.

Visual confirmation:

- **t=0.70 screenshot**: grass reads olive-warm, sand reads
  tan-orange, water teal — the whole scene carries a sunset
  cast.
- **t=0.99 screenshot**: grass reads deep-green-navy, water
  inky, stars visible. Classic night cool-blue.

The two scenes *look like different times of day* now, not
the same blue-cast world at different darknesses. Which was
exactly 011's prescription.

## what I noticed (graphics)

1. **No boundary pop at t=0.10.** The transition from dawn
   rose to the dusk/night lerp happens at t=0.10, where
   `daylight` has hit 1.0 and the overlay is *off*. So the
   profile switch is invisible — the overlay turns on
   already in the lerp branch. Free win from existing
   architecture.

2. **The lerp from amber → indigo crosses near-grey around
   t=0.85.** That's when the two profiles balance out
   (halfway through the blend). The midpoint reads as
   low-chroma cold grey, which doesn't feel odd — it reads
   as "the warm is fading, night hasn't fully set in."
   Happy accident.

3. **Winter tileShift interacts differently now.** 010
   reported winter felt over-cold. Now the tint isn't
   stacking more blue on top of an already-blue tileShift —
   at dusk the warm amber *opposes* the winter-cool tile
   cast, pulling it toward neutral. Should make winter
   sunset readable; deep winter night still reads cold
   (both tint and tileShift pull blue). A future
   `the-photographer` tick could verify this in winter.

4. **Dawn is short.** The `t < 0.10` window is only 4
   simulated hours and most of it has low darkness (the
   ramp from 0.55 to 1.0 happens in the first 10% of day).
   So the rose cast is fleeting but present. Good — dawn
   is supposed to be brief.

## what I did NOT do

- Touch `getDaylight()`. Per 011, not re-shaping the curve.
- Add a `getTintColor(t)` function. Considered it — would
  be cleaner architecturally and enable a `tintCurve`
  sampler — but the change was meant to be surgical. The
  inline logic is 12 lines; promoting to a module export
  is the right move only once a second call site wants
  the same data (e.g., a photo-mode, or a tintCurve debug
  sampler). Filed as `011 idea` — still open.
- Touch the `SEASONS[*].tileShift` values. 010 suggested
  winter was over-cooled; the hue-variation fix *may*
  solve this incidentally (by warming dusk) without
  needing a tileShift change. A future play tick should
  verify before anyone touches tileShift.
