## finding 1 (HIGH): lastDeathDay tracker missing at 2 death sites

**The pattern:** `G.stats.citizensDied++` and
`G.lastDeathDay = G.day` should travel
together. They don't.

**Audit by grep:**
```
js/combat.js:134     stats.citizensDied++
js/combat.js:135     lastDeathDay = G.day  ✓
js/economy.js:???    stats.citizensDied++ (starvation block)
js/economy.js:351    lastDeathDay = G.day  ✓
js/events.js:278     stats.citizensDied++ (PLAGUE)
                     lastDeathDay NOT SET   ✗
js/soldiers.js:145   stats.citizensDied++ (BATTLE)
                     lastDeathDay NOT SET   ✗
```

**Two missed sites:**
1. `events.js:278` — plague event kills
   citizens in a loop, increments
   `citizensDied` per death, but doesn't
   touch `lastDeathDay`.
2. `soldiers.js:145` — soldier death from
   combat increments `citizensDied`, doesn't
   touch `lastDeathDay`.

**Real-world impact scenario:**
A realm experiences:
- Day 50: plague kills 5 citizens.
  `citizensDied = 5`, `lastDeathDay` still
  undefined (or stale).
- Day 51-150: 100 raid-free + starvation-free
  days. No combat/economy deaths.
- Day 151: `no_death_known` beat trigger
  evaluates:
  - `stats.citizensDied >= 1` → true ✓
  - `lastDeathDay !== undefined` → may be
    false (if no prior raid/starvation death),
    blocking the beat.
  - OR if a previous starvation set it to
    day 5, `(151 - 5) >= 100` → true →
    beat fires WRONGLY (because plague at day
    50 should have reset the counter).

**Severity HIGH.** The 228 sustained-state
beat produces incorrect output for realms
that experience plague or military combat
deaths. The 233 HUD life-streak indicator
similarly misreports.

**Recommended fix (242):** add
`G.lastDeathDay = G.day;` to both sites.
2-line patch:
```js
// events.js:278 (inside plague death loop)
G.stats.citizensDied = ...
G.lastDeathDay = G.day;  // ADD

// soldiers.js:145 (inside soldier death loop)
G.stats.citizensDied = ...
G.lastDeathDay = G.day;  // ADD
```

## finding 2 (MEDIUM): `_WINTER_CAPS` may be off post-239 sprite resize

The 239 size bump (~20%) increased SVG sprite
heights by 6-12px. `_WINTER_CAPS` cap geometry
(per 094 invariant) was tuned for the OLD
sprite era — caps positioned via fixed pixel
offsets relative to s.y.

**Effect:** caps may now sit slightly off-
position on the new larger sprites — likely
1-2 pixels low or floating above the new tops.

**Severity MEDIUM** because it's cosmetic, not
correctness. 218's verify-phaseb winter
screenshots may need a re-shoot to confirm
parity. Filed at 239; not a NEW finding.

## finding 3 (LOW): sprite cache could leak across many kingdoms

220 introduced per-kingdom variant caching:
`${type}__${kname}` keys. A user playing 50
distinct realm-names in a single session would
load 11 × 50 = 550 cached `<img>` elements
(each holding a Blob URL of variant SVG text).

**Severity LOW** — typical play sessions
don't switch kingdom names that often.
Browser eventually GCs unused Blob URLs.
Filed at 220 originally.
