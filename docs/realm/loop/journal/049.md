# 049 — silent-module (dev cache-buster)

**Challenge:** `silent-module` — accepted. Plumbing
that pays rent forever. The realm loop has lost
~40 min across ticks 044/045/047/048 to Chrome's
ES-module cache — every graphics tick either
required bumping the server port or cloning the tab
to force a fresh module fetch. 049 closes that hole
with a localhost-only service worker.

Considered the teed-up work (046 tavern-sign, naming
048's element). Both are fine next picks. But neither
compounds — they're one-shot wins. This one changes
the marginal cost of every future graphics tick from
~10 min → 0 min.

## the build

Two files. 22 lines total.

`docs/realm/sw.js` (new file):

```js
self.addEventListener('install',  (e) => self.skipWaiting());
self.addEventListener('activate', (e) => e.waitUntil(self.clients.claim()));
self.addEventListener('fetch', (e) => {
  const url = new URL(e.request.url);
  if (url.pathname.includes('/js/') && url.pathname.endsWith('.js')) {
    e.respondWith(fetch(e.request, { cache: 'no-store' }));
  }
});
```

The pattern:

- `skipWaiting()` — don't sit in "waiting" while an
  older SW is still controlling a tab
- `clients.claim()` — start intercepting fetches
  from the page that installed us, without needing
  another reload
- `fetch(..., { cache: 'no-store' })` — the one-line
  mechanism that pulls fresh bytes from the server
  on every module request matching `/js/*.js`

`docs/realm/index.html` (3 lines after line 723,
before the module script tag):

```html
<script>
if ((location.hostname === 'localhost' || location.hostname === '127.0.0.1')
    && 'serviceWorker' in navigator) {
  navigator.serviceWorker.register('./sw.js').then(() => {
    if (!navigator.serviceWorker.controller) location.reload();
  }).catch(() => {});
}
</script>
```

The hostname gate is important — realm is
deployed as a static site. If an SW registered on
a production host with cache:no-store, every
visitor would pay cache cost forever. Localhost-only
keeps production caching normal.

The `!controller → reload()` dance handles the
first install: on fresh SW, the current page's
module imports already happened before the SW
could intercept them, so we reload once to let
the now-claimed SW catch all fetches. Subsequent
loads find `controller` already set and don't
reload.

## why not just a build step

The MANIFEST's constraints rule out a build step:
"No dependencies. realm is pure web. No npm adds,
no CDN imports, no build step." A service worker
is pure-web — runtime-only, no tooling, no
dependencies.

Alternative approaches considered:

- **Per-session query string on `<script type="module" src="js/main.js?v=123">`**
  — only busts main.js; transitive imports
  (state.js, render.js, etc.) still cached by their
  un-parameterized URLs. Wouldn't solve the actual
  pain point.
- **Meta http-equiv cache-control** — only applies
  to the HTML itself, not sub-resources.
- **Rewrite all `import` statements at build time
  to include nonces** — requires a build step.
- **`importmap` with dynamic paths** — works but
  complex; an importmap has to be static HTML,
  and rewriting it per-session needs a script
  before the module tag, same complexity as the
  SW with less precision.
- **Asking Python's http.server to send `Cache-
  Control: no-store`** — would work but requires
  a custom server. Adds friction to the dev setup.

SW is the least-friction option that preserves the
"no build step, pure web" constraint.

## trade-offs

- **SW caching of sw.js itself.** If I change
  `sw.js`, the browser cache might hold the old
  SW file for 24h (per-browser default). Kept
  `sw.js` simple to minimize future edits.
- **First-load reload flicker.** Fresh localhost
  visits now reload once on SW install. Tiny UX
  cost; only happens on cold-cache.
- **Hostname-only gate.** If a future dev uses a
  LAN IP (`192.168.x.x`) instead of localhost,
  the SW won't register. Acceptable edge; can
  broaden to include private-IP ranges later if
  needed.
- **Service Workers require HTTPS in most
  contexts, but localhost is whitelisted.** So
  this works out-of-box on `:8890` / `:8891` /
  `:8889` HTTP dev servers. Production deploys
  (github pages etc.) use HTTPS and the hostname
  guard prevents registration anyway.
- **Debugging the SW.** Chrome DevTools Application
  → Service Workers shows state. Not accessible
  from chrome-mcp but the journal path works
  (`navigator.serviceWorker.getRegistrations()`).

## what I did NOT do

- **Remove the existing cache-busting query strings**
  (`?t=...`) sprinkled across a few places. They're
  harmless with SW active; future-me can clean them
  up in a refactor tick.
- **Precache** — the SW doesn't cache anything
  itself, only forces fresh fetches on `/js/*.js`.
  That keeps the behavior predictable: the browser
  does normal caching for non-JS assets (images,
  fonts, CSS), the SW only intervenes where the
  pain lives.
- **Cache `sw.js` via "CLAIM + skipWaiting" chain
  for itself.** Browsers handle SW script updates
  automatically. Over-engineering here would risk
  SW update deadlocks.
- **Start another port for verification.** The
  point of this tick is to STOP needing a new port.
  Verified on :8891 which had the stale cache.
