## the build

`docs/realm/js/enhancements.js:4694-…` —
the existing single-prose-per-outcome
raid handler is replaced with variant
pools. Each of the 5 outcome shapes
(razed / losses-only / mixed / victory /
bloodless) gets 4 prose variants chosen
by `day % pool.length`.

```js
const _RAID_PROSE_POOLS = {
  razed: [
    (day) => `Raid on day ${day}: the village was razed. None remain to tell it.`,
    (day) => `Raid on day ${day}: the realm fell silent. The fires went out one by one.`,
    (day) => `Raid on day ${day}: the realm did not survive the night. There is no one left to bury the dead.`,
    (day) => `Raid on day ${day}: the raiders left only smoke and stone.`,
  ],
  // ... 4 more pools, 4 variants each
};
export function _pickRaidProse(day, kills, losses, popAlive) {
  let pool;
  if (popAlive === 0) pool = _RAID_PROSE_POOLS.razed;
  // ... etc
  const idx = ((day % pool.length) + pool.length) % pool.length;
  return pool[idx](day, kills, losses);
}
```

20 prose lines (5 pools × 4 variants) +
~10 LoC selector helper. Net +60 LoC in
enhancements.js.

## why this shape

- **Pure-function helper, exported.**
  `_pickRaidProse(day, kills, losses,
  popAlive)` is testable from outside
  without driving the full game loop.
  Pure → no global state, no side
  effects.
- **First variant per pool = original
  prose.** Preserves backward
  compatibility for any save/test that
  relied on exact text. Day 1 raids
  still see "the village was razed.
  None remain to tell it." as variant 0.
- **`day % pool.length` selection.**
  Deterministic + reproducible on save/
  load. RNG would create save-load
  divergence (chronicle entries differ
  between sessions); day-based hash
  doesn't.
- **Safe modulo `((day % n) + n) % n`.**
  Guards against negative day values
  (shouldn't happen but cheap insurance).
- **4 variants per pool.** With
  `day % 4`, consecutive raids cycle
  through all 4 before repeating. Player
  watching successive raids on days 10/
  11/12/13 sees 4 distinct prose lines.
  More than 4 risks variants reading
  too similar; fewer than 4 risks
  cycling visible.
- **5 outcome shapes preserved.** The
  outcome-shape selection logic
  (popAlive===0 / losses>0&&kills===0
  / etc) is unchanged from 010's
  original logic. Only the prose
  generation per shape varies.
- **No new authoring system.** No new
  cluster, no new tag, no new schema
  field. Just expanded prose pools
  inside the existing raid handler.
  Smallest-possible-change for the
  observed problem.

## why this surprises (post-fix)

For a long-lived realm with 30
successful defenses (all-victory shape),
pre-264 saw "Raid on day N repelled —
K foes slain, none of ours lost." × 30.
Post-264, the same 30 raids cycle
through 4 distinct prose lines:
- "Raid repelled — N foes slain, none
  of ours lost."
- "N raiders cut down at the gate. The
  village slept untroubled."
- "The raiders broke at first light. N
  of them never made it home."
- "N foes ran into the realm's sword
  and stayed. We took no losses."

Each variant has the same INFORMATION
(N foes, no losses) but different
register: military report / pastoral
peace / prose-rhythmic / declarative.
Reading the chronicle now feels like
reading a chronicler with VOICE rather
than a templated combat log.

## what I did NOT do

- **Add a 5th or 6th outcome shape.**
  The existing 5 cover the state space
  reasonably. Could add (e.g.) a
  "raiders demanded tribute and got it"
  or "raid lasted two days" but those
  are NEW MECHANICS not just prose
  variants.
- **Vary prose by KINGDOM (kingdom-hash).**
  Considered. Day-hash is sufficient
  for the within-realm variety problem;
  kingdom-hash would add inter-realm
  variety but reduces shared culture
  ("every realm sees same prose for
  the same outcome on the same day"
  has a small charm).
- **Vary prose by RAID NUMBER (1st /
  2nd / Nth).** First raid feels
  different from 30th, but adding a
  raid-counter dimension makes
  authoring more complex and was not
  the 260 finding.
- **Promote variant prose to invariant.**
  Per 231/251/257 conservative-
  promotion. First use of "outcome-
  shape × variant-pool" pattern; stays
  observed-pattern.
