## emergent patterns — earned during Phase A

These weren't in 171's plan but emerged from
authoring 5 sprites in succession:

### 1. The "interior is OPEN" idiom

Tavern (175): warm hearth glow + slightly-ajar
warm door light + cobble path + bunting +
hanging harvest bundle.

Bakery (180): warm light spilling under door +
oven-warmth window + dough-smoke wisps + bread
paddle leaning + flour dust.

Market (179): no walls/door at all + bunting +
goods on counter + coin pile.

**Pattern: each "commercial / hospitality"
building reads as ACTIVELY OPEN via warm-light
secondary signals + tools/goods visible
externally.** Future SVG sprites for
public-facing buildings should include similar
"OPEN" cues. Filed as an invariant for Phase B
integration to preserve.

### 2. The "tools propped/hanging" idiom

Blacksmith (176): iron tongs hanging on wall +
hammer mid-strike on anvil + propped hammer
against wall.

Bakery (180): bread paddle leaning against wall.

Market (179): barrel + stacked crates beside
stall.

**Pattern: working buildings show their tools.**
Counters or industrial buildings get visible
trade implements. Future SVG buildings
(barracks?) should follow when applicable
(weapons rack? guard's helmet?).

### 3. Family palette correspondences

Domestic warmth family (house / tavern / bakery)
share: warm wood walls / plaster + half-timber
posts + warm interior glow + gold trim. Built
out idiom-by-idiom over 173/175/180.

Defensive / industrial family (tower / castle /
blacksmith) share: stone block patterns + flag/
crenellation/parapet + cool greys + iron
hardware.

Sky / civic family (church / constellation
visual): gothic-arch language + stone block
patterns + religious / celestial palette.

Exception: granary, windmill, market are
**hybrid** — each blends elements but holds a
distinct silhouette identity.

**Phase B per-instance variation should respect
these families.** A church variant should stay
in gothic-arch language; a house variant should
stay in domestic-warmth language; etc.

  render-layers.md. No "almost-complete"

## refreshed integration-concern checklist

171 / 165 listed 6 concerns. Phase A
authoring revealed which are heavier than
estimated:

| Concern | 171 estimate | Phase A finding | Updated weight |
| ------- | ------------ | --------------- | -------------- |
| Day/night tint | medium | unchanged — same multiply-blend approach | medium |
| Winter caps | medium | 158's dome-conforming arc reads particularly well; SVG must compose UNDER it | medium |
| Hover halo | low | low — straightforward | low |
| Fog modulation | low | per-frame opacity; straightforward | low |
| **Per-instance variation** | medium | **HEAVY** — 5 of 10 sprites have variants; 70+ configurations to handle | **heavy** |
| **Animation** | low | **MEDIUM** — 4 sprites have canvas animations; not "just one castle flag" | **medium** |

**New finding (not in 171):**

- **Loading strategy.** Lazy-load via
  `<img src="./assets/sprites/X.svg">`? Inline
  via fetch + `innerHTML`? Sprite atlas? Each
  has tradeoffs for the canvas pipeline.
  Filed for Phase B step 1 (scaffolding)
  decision.

- **Z-order.** Canvas drawX calls layer
  buildings in render order; SVG sprites at
  the same z-level need a different
  composition path. 158 winter cap renders
  AFTER drawGranary; SVG winter cap must
  render AFTER svg granary. Filed.

- **Picking / hover detection.** Canvas
  hover uses pixel-perfect or bounding-box
  hit detection on the canvas. SVG `<img>`
  doesn't expose pixel data the same way.
  May need fallback bounding-box only.
