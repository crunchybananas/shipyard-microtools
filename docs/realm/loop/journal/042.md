## the fix

**One JS function + one HTML element + one CSS block**
across two files.

`docs/realm/js/ui.js` (inside `toggleResearchPanel`):

```js
if (!isOpen) {
  renderResearchPanel();
  _wireScrollCue(panel);  // NEW
}

function _wireScrollCue(panel) {
  const c = panel.querySelector('.research-content');
  if (!c) return;
  const update = () => {
    const atBottom = c.scrollTop + c.clientHeight >= c.scrollHeight - 4;
    const canScroll = c.scrollHeight > c.clientHeight;
    panel.classList.toggle('has-more-below', canScroll && !atBottom);
  };
  if (!c._scrollCueWired) {
    c.addEventListener('scroll', update, { passive: true });
    c._scrollCueWired = true;
  }
  requestAnimationFrame(update);
}
```

- Toggles `body has-more-below` class on `#research-panel`
  when content is scrollable AND user isn't at the bottom
- Uses `rAF` for the initial check so layout has settled
  post-renderResearchPanel
- Idempotent: repeat-opens don't double-bind the scroll
  listener (guarded by `c._scrollCueWired`)

`docs/realm/index.html` (CSS):

```css
#research-panel.has-more-below::after {
  content: '';
  position: absolute;
  left: 1px; right: 1px; bottom: 1px;
  height: 2rem;
  pointer-events: none;
  background: linear-gradient(to top, var(--panel) 20%, rgba(0,0,0,0) 100%);
  border-radius: 0 0 11px 11px;
  z-index: 1;
  animation: scrollCueFade 0.2s ease;
}
#research-panel.has-more-below .rp-scroll-hint {
  display: block;
}
.rp-scroll-hint {
  display: none;
  position: absolute;
  bottom: 0.35rem; left: 50%; transform: translateX(-50%);
  font-size: 0.7rem;
  color: rgba(255,255,255,0.55);
  pointer-events: none;
  z-index: 2;
  letter-spacing: 0.04em;
}
@keyframes scrollCueFade { from { opacity: 0 } to { opacity: 1 } }
```

`docs/realm/index.html` (DOM): added `<div class="rp-scroll-hint">more below ▾</div>`
inside `#research-panel`.

Two visible cues layered:
- **Soft gradient** at the bottom 2rem — fades the last
  row of content so it's clearly not the end
- **"more below ▾"** text hint at 0.35rem from the
  bottom, subtle gray

Both appear/disappear together via the `.has-more-below`
class.

## why this approach

Considered and rejected:

- **Pure CSS mask on `.research-content`** — would
  always fade the bottom even when at end. Persistent
  unwanted ghost at panel bottom.
- **JS scroll-to-bottom detection on every tick** —
  works but fires constantly; wasteful vs one-shot
  event listener.
- **Show gradient on ALL 5 panels using
  `.research-content` class** — out of scope for this
  tick per 040's HIGH scoping. Other panels
  (Chronicle, Stats, Achievements, Notifications,
  Pop) have the same scroll-cue gap; that's a
  follow-up tick using the same `_wireScrollCue`
  helper. Filed.

## what I did NOT do

- **Propagate the cue to the other 4 panels**
  (Chronicle / Stats / Achievements / Notifications
  / Pop). All use `.research-content`; same class of
  bug. Filed as 042 follow-up.
- **Animated pulse on the hint** — was tempting but
  static is fine. Animation could draw too much eye
  for a passive hint.
- **Paginate the research tree** — a bigger UX
  change that competes with the scroll-cue. If 040's
  sparse-tier cleanup happens, paginating may become
  unnecessary. Filed as 040 sparse-tier follow-up
  (already listed).
