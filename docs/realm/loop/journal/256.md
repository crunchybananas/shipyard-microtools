## findings

### HIGH — Mayor mechanic categorization mismatch in 6/6 narrative

The "6-of-6 named-character mechanic cast"
narrative (243 commit, 251 doc, 252/253
journal) groups SIX heterogeneous mechanic
shapes under one count:

| Character | File:Line | Shape | Effect |
|-----------|-----------|-------|--------|
| Teacher   | tech.js:199 | multiplicative continuous | +10% research forever |
| Merchant  | trade.js:24 | multiplicative continuous | +5% trade forever |
| Smith     | combat.js:216 | multiplicative continuous | +5% fire-rate forever |
| Bard      | economy.js:324 | additive baseline continuous | +5 happiness baseline forever |
| Rival     | economy.js:442 + combat.js:108 | event-trigger paired | +10% raid count + +5g/raider |
| **Mayor**   | **tech.js:141** | **structural-unlock one-time gate** | unlocks `townhall` (state.js:77) |

The first 5 deliver continuous numeric
effects every tick the realm runs. The 6th
(mayor) is a **gate that unlocks a building**.
Once townhall is built, the mayor has zero
ongoing mechanic effect — the townhall does.

This isn't WRONG (243 acknowledged it as
"4th shape: structural-unlock"). But the
6/6 cast framing in 243/249/251/252/253
journals reads as if the six shapes are
comparable. They are not. A future audit
could legitimately conclude "we shipped 5
continuous bonuses and 1 building-unlock,
not 6 of the same kind."

**Recommendation:** narrative-surfaces.md
named-character-mechanic invariant should
explicitly NOTE that mayor's shape is
categorically different — not just "4th
shape" mentioned in passing but a top-line
note that 5/6 are continuous and 1/6 is a
gate. This protects future authors from
copy-pasting the cast-completion frame
when adding a 7th character.

### MEDIUM — Townhall has no uniqueness constraint; pop:5 stacks linearly

`state.js:77` defines townhall with
`cost:{stone:40,wood:30,gold:20}, happiness:8,
radius:6, pop:5` — but no `maxCount` /
`unique` / `isUnique` field.

`isBuildingUnlocked('townhall')` (tech.js:141)
checks only mayor existence; it does NOT
check existing townhall count.

`economy.js:66` adds `def.pop` linearly to
`G.maxPop` per build. Build N townhalls →
+5N global max population.

For comparison: castle has happiness:20 + pop:10
+ defense:50, also no uniqueness — but castle
is end-game (most realms build 1). Tavern
(happiness:10 radius:5) + church (happiness:15
radius:6) are similarly stackable.

**This is CONSISTENT with the existing pattern.**
But: the named-character-gated framing in 243
("mayor → unlocks the townhall") implies a
one-shot reward. If the design intent was
"named-mayor unlocks a unique civic structure,"
a `maxCount: 1` constraint would match the
intent. If it was "mayor enables a building
tier," current design is fine.

**Recommendation:** clarify intent in
state.js:77 comment or BUILDING_FIRST_BEATS
gate. If maxCount:1 was intended, ship it as
a one-line fix:

```js
townhall: {
  ..., maxCount: 1,  // 256 audit: structural-unlock
                     //   shape implies one-shot, not stackable.
}
```

(Would require corresponding check in the
build dispatch path; current code may not
respect maxCount yet — needs verification.)
