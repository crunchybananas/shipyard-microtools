# 044 — time-aware (dream lens)

**Challenge:** `time-aware` — accepted. Never picked
in 43 prior ticks. Evolution mandate favored a fresh
surprise slot after 043's NIGHTMARE shipped, and
`time-aware` compounds most cleanly with the existing
dream system (039) — making realm's newest content
surface respond to an ambient input the game never
listened to before: the clock on the wall behind the
player.

## the build

One new exported function (`_realWorldDreamLens`),
one hook into `checkDreamBeat`, one debug entry
(`G.debug.dreamLens`). Touches `js/story.js` and
`js/main.js`.

The lens consults `new Date()` once and returns:

```
{ boost: [threadName, threadName], special: { intro, closer } | null }
```

### seasonal boost

Doubles the weight of two dream threads per season
(by pushing them into `avail` a second time before
hash-picking):

| Months       | Season | Boost               |
| ------------ | ------ | ------------------- |
| Dec, Jan, Feb | winter | `hearth` + `warning` |
| Mar, Apr, May | spring | `harvest` + `water`  |
| Jun, Jul, Aug | summer | `water` + `market`   |
| Sep, Oct, Nov | autumn | `warning` + `forge`  |

Critically, the boost only fires for threads *already
available* in this realm — we don't invent threads
that don't apply. An early-game realm with no farms
doesn't get `harvest` forced in just because it's
April.

### special-date overrides

Five check-in-order cases replace the dream's
`"At dawn the realm wakes..."` intro with something
that belongs to the day:

- **Oct 31** — *"On the thinnest night of the year..."*
  closer *"Tomorrow the veil thickens again."*
- **Dec 20–22** (winter solstice) — *"On the longest
  night..."* closer *"The dark will turn now."*
- **Jun 20–22** (summer solstice) — *"On the shortest
  night..."* closer *"The light will turn now."*
- **Mar 19–21** (spring equinox) — *"On the balanced
  night..."* closer *"The seasons change hands."*
- **hour === 0** (midnight local time) — *"At the
  witching hour..."* closer *"No one speaks of them,
  but everyone remembers."*

First match wins. Only fires when `checkDreamBeat`
fires (day 14 + every 14 days, at dawn, once per
game-day).

## why this shape

039's dreams were pure content-from-state. Kingdom
+ day + what you've built → what you dreamed. That
determinism is load-bearing: two people running the
same save get the same dream, and screenshots mean
the same thing.

044 introduces an *ambient* input — the real calendar
— that BREAKS cross-session determinism by design.
Same save, same day, different month → different
flavor. This is the *point* of the `time-aware`
challenge: the game listens to something the player
isn't aware it listens to.

The determinism bound:
- Within a single calendar **day**, same realm + same
  day → same dream (seed unchanged).
- Across calendar days/months, dreams diverge. But the
  `dream_${G.day}` flag makes this a one-time event —
  you can't "re-roll" it by time-traveling unless you
  also clear `storyFlags`.

So in practice: whichever real-world day the player
*first* opens dawn on day 14, that's the dream they
get — stamped with the season they were in when the
game noticed.

## what I did NOT do

- **Calendar events on top of 039's dream timer.** I
  didn't add "a dream that fires only on Halloween
  regardless of G.day." That would be `a-scene-that-
  happens-once`-adjacent and changes the dream
  schedule. The lens rides on the existing 14-day
  cadence — stays subtle.
- **Location-aware inputs** (latitude-aware season
  inversion for southern hemisphere players, etc.).
  `new Date()` is browser-local, good enough.
- **Persist the lens state.** Each dawn the lens runs
  fresh; no saved "dream was biased by X month" flag.
  If the realm is played across seasons, later dreams
  naturally reflect whatever month they fire in.
- **Propagate the lens to chronicle beats.** The lens
  is dream-only for now. Could plug into
  `checkNightmareBeat` (043) — a Halloween nightmare
  would write itself, but that shortens the surprise
  horizon for nightmares. Filed instead.
- **Touch the UI.** No visual indicator that the
  dream was "time-seasoned." The beat is felt, not
  announced. Evolution-mandate shape.

## trade-offs noted

- **Non-determinism is a feature here, not a bug.** A
  future tick that wants "given a kingdom + day, show
  the canonical dream" (e.g. for a lore-export view)
  will have to explicitly call `_realWorldDreamLens`
  with a fixed seed date. Documented in the function's
  header.
- **Boost strength is fixed at 2×.** Could be tuned
  (e.g. summer = 3× water for a "drowned-in-water-
  dreams" feel). Left as default for now; 40-realm
  sample shows 2× already produces a clearly readable
  skew without overwhelming other threads.
- **Hour-0 witching check is checked last.** If you
  load at midnight on Halloween, Halloween wins. This
  is the correct precedence (holidays beat recurring
  clock states), but worth noting.
