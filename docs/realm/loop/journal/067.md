## the fix

`docs/realm/js/render.js`:

1. New exported `applyNightTint(ctx, daylight,
   dayPhase, dayLength)` near the top of the file,
   before `render()`. Preserves the exact 012 + 017
   math (dawn rose / dusk amber → indigo night,
   capped at 0.7 darkness).
2. `render()` at ~2515 replaces the 35-line inline
   tint block with one call:
   `applyNightTint(ctx, daylight, G.dayPhase, G.dayLength)`
3. `renderBuildingIsolated` accepts a new
   `opts.dayPhase` parameter (defaults to
   `dayLength * 0.5` = midday). After
   `drawBuilding` completes, it calls
   `applyNightTint(octx, daylight, dayPhase,
   dayLengthVal)` so the isolated canvas gets
   tinted too.

Args are explicit (not pulled from G inside the
tint function) so offscreen callers don't need to
swap G state for tint purposes.

**Net LoC**: new function ~35 lines, inline block
removed ~35 lines, helper integration +~5 lines.
Net ~+5.

## why this shape

Considered:

- **Duplicate the tint math inside the helper**
  (066's quick-fix alt). Works but creates two
  copies of non-trivial color math. If a future
  tick tunes 012's hue-lerp, it would need to
  update both.
- **Move the tint into a post-render enhancement
  hook** (registerScreenRenderer). Would decouple
  nicely but adds ordering complexity: the tint
  must apply BEFORE screen-space UI overlays,
  AFTER all world sprites. The current inline
  placement is already correct; a registry would
  just re-assert that ordering in config.
- **Keep tint in G-aware form** with no args.
  Rejected: offscreen contexts need to set G
  state for the tint to read, which we'd have to
  restore. The explicit-arg form is cleaner.

Explicit args + shared function is the lightest
refactor that closes 066's coverage gap.

## verification (chrome-mcp on :8891 via 049 SW)

Castle palette through the helper at 4 different
phases:

```
BEFORE 067 (helper always midday):
  castle (any daylight):  ~#808060 | #606040 | #606060

AFTER 067:
  midday (dl 1.0):       #808060 | #606040 | #606060   ← untinted
  dawn   (dl 0.4, p=792): #406020 | #806040 | #408020  ← rose-dimmer
  dusk   (dl 0.5, p=2808):#208020 | #408020 | #806040  ← amber-shift
  night  (dl 0.25, p=180):#206020 | #406020 | #806040  ← indigo-darker
```

**Midday palette is identical pre/post 067** (no
tint applied when `daylight >= 0.95`). Dawn/dusk/
night palettes each show distinct shifts matching
012's hue-lerp profile:
- Dawn R drops least (rose bias)
- Dusk has mid amber (`#806040` surfaces on the
  warm side)
- Night R drops most aggressively (cool indigo)

The `#808060 → #206020` shift at deep night is ~4×
darker R, matching `cappedDarkness * kR` at
daylight=0.25 → darkness=0.75 → cap at 0.7 × kR=180
→ tintR=255-126=129 ≈ multiply with castle's
R(~128)=64, which quantizes to #20. ✓

Main render verification: `render()` now calls
`applyNightTint(ctx, daylight, G.dayPhase,
G.dayLength)`. Since the extracted function is
bit-identical to the old inline block's math, the
main game render pass is byte-for-byte unchanged.
Visually confirmed the game still renders at startup
