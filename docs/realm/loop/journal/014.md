## the bug

010 observed: survive-5-raids counter went 0→3/5 in a
session where the player had zero military buildings.
"counts raid-fired, not defense-succeeded." The mission is
passable by standing still.

Root cause, found today:

`js/main.js:281-284` — at every day-tick, snapshot
`G.nextRaidDay`, run `checkRaids()`, then if
`G.nextRaidDay` advanced, increment `raidsSurvived`:

```js
const _raidDayBefore = G.nextRaidDay;
collectTaxes();
checkRaids();
if (G.stats && G.nextRaidDay > _raidDayBefore) G.stats.raidsSurvived++;
```

But inside `checkRaids()` (`economy.js:360-429`), the
sequence is: warn → spawn enemies → ... → push
`G.nextRaidDay` forward by `raidInterval`. So
`nextRaidDay` advances at the *moment of spawn*, not
after the raid resolves. The old increment was running on
raid-START, regardless of defense.

## the fix

Single logical change split across two files:

1. **`js/main.js:281-284`** — removed the unconditional
   day-tick increment. Replaced with a comment pointing
   to the new site.
2. **`js/economy.js:427-430` (approx)** — inside the raid-
   fire block, after `G.nextRaidDay` is updated, added a
   gated increment reusing the existing local
   `hasDefense` variable:

```js
if (G.stats && hasDefense) G.stats.raidsSurvived++;
```

`hasDefense` was already computed at line ~371 for raid-
scaling purposes:

```js
const hasDefense = G.buildings.some(b =>
  b.type === 'wall' || b.type === 'barracks' ||
  b.type === 'tower' || b.type === 'archery' ||
  b.type === 'castle'
);
```

Reusing it keeps the survival-counting and raid-scaling
aligned: same definition of "defended" for both axes.

Both files pass `node --check`.

## verification

Chrome-mcp, fresh New Game (Peaceful, Easy), no buildings
placed. Forced raids via console:

**Test 1 — defenseless raid**:
```
before: day=1, nextRaidDay=8, raidsSurvived=0, defBuildings=0
(jumped day to 8, fired checkRaids())
after:  day=8, nextRaidDay=16, raidsSurvived=0, defBuildings=0, enemies=2
        → bugFixed: true  (raid spawned but was NOT counted)
```

**Test 2 — defended raid**:
```
(added a Barracks to G.buildings)
before: raidsSurvived=0, defBuildings=1
(jumped day to next raid, fired checkRaids())
after:  raidsSurvived=1, defBuildings=1
        → incremented: true  (raid counted with defense)
```

Both directions behave correctly.

## what I noticed

1. **The fix is minimal because the raid-fire already knew
   about defenses.** `hasDefense` was being computed for
   unrelated scaling logic. The fact that `raidsSurvived++`
   lived in a different file from that computation is the
   architectural pattern that let this bug exist —
   side-effects-at-a-distance. Moving it into the same block
   where `nextRaidDay` is actually advanced and where
   `hasDefense` is visible tightens the invariants.

2. **"Defense at spawn-time" is not identical to "actually
   survived."** If a player has 1 Barracks that gets
   destroyed before soldiers intercept any raider, my fix
   still counts it as survived. A stronger fix would wait
   for all enemies to die/leave, then check damage. Scope
   for a future tick if the-gamer plays long enough to
   catch the edge case. 010's complaint (0→3 with zero
   defenses) is structurally resolved; the edge case is an
   improvement, not another bug.

3. **Existing first-raid logic stays correct.**
   `economy.js:370` reads `const isFirstRaid = !G.stats?.raidsSurvived`
   — that "is this my first raid?" check still works
   because `raidsSurvived` starts at 0 and my fix only
   increments it when defenses exist. Pre-defense players
   will see the same "first raid = 2 raiders" behavior
   every raid until they build something military, which
   matches the intent of a gentle first-touch.
