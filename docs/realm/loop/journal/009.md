## what

`render.js:865-880` drew the valid-tile highlight as a single
fill pass with alpha pulsing between 0.06 and 0.30, color
`#22c55e` (Tailwind green-500). Against bright grass tiles
(which are also green), the fill blended into the terrain —
007 and earlier ticks both missed it on first glance.

Three changes in place:

1. **Pulse range**: `0.18 + 0.12 * sin()` → `0.28 + 0.14 * sin()`.
   Alpha now ranges 0.14–0.42 instead of 0.06–0.30 — a roughly
   40% uplift at peak. Still pulsing, still "gentle."
2. **Fill color**: `#22c55e` → `#4ade80` (green-400). Brighter,
   slightly less saturated; reads more like "highlight" and
   less like "grass."
3. **New stroke**: `strokeStyle='#bbf7d0'` (green-200, near-white
   pale green), `lineWidth=1`, alpha `min(1, pulse * 2)` so the
   edge is always roughly 2× the fill alpha. This is the
   structural fix — an edge defines the tile even when the fill
   fights the background.

Inline comment cites the loop + the 007 finding so a future
tick reviewing this can trace the reasoning.

## what I noticed (graphics)

**Mild busy-ness concern**: when Farm is selected and most of
the visible map is grass, dozens of tile edges glow at once.
That reads as "the map has a grid" which is accurate — realm
is tile-based — but it's visually active. In practice this is
fine because:
- The grid appears only when a building is selected (player
  is actively in build mode, not at rest)
- The pulse is still gentle enough not to fatigue the eye
- Mountain/forest/water silhouettes break up the grid pattern
  naturally

If a later tick argues the grid is *too* busy, the stroke
alpha is the cheap lever — drop to `min(1, pulse)` or
`min(0.7, pulse * 1.5)` for softer edges.

**Side benefit**: the grid now teaches map layout. A novice can
see what counts as "one tile" at a glance. Prior to this fix,
the tile boundaries were only visible during hover (one tile
at a time). Now the whole buildable region reads as a discrete
grid when you're in build mode.
