## the change

`docs/realm/js/main.js:startNewGame` — 3 changes:

### 1. Import `setSeed`

```js
import { ..., setSeed } from './state.js';
```

One-token addition to the existing import.

### 2. Reorder: read kingdomName BEFORE generateWorld

Previously the flow was:
1. generateWorld() — uses time-seed
2. (later) read kingdom name from input
3. beginGame()

Now:
1. read kingdom name
2. hash it → setSeed
3. generateWorld() — uses kingdom-seed
4. (everything else)

### 3. Inline hash function + setSeed call

```js
const nameInput = document.getElementById('kingdom-name-input');
G.kingdomName = (nameInput && nameInput.value.trim()) || 'Realm';
let kh = 0;
for (let i = 0; i < G.kingdomName.length; i++) {
  kh = ((kh << 5) - kh + G.kingdomName.charCodeAt(i)) | 0;
}
setSeed(Math.abs(kh) || 1);
generateWorld();
```

The hash mirrors the `_dreamHash` shape used throughout
story.js — same algorithm, same output per kingdom
name. Done inline rather than importing `_dreamHash`
(story.js's internal helper) to avoid cross-file
coupling for 4 lines.

The `|| 1` fallback handles the edge case where a
kingdom name hashes to exactly 0.

## why this shape

- **Minimum-viable fix.** 131 HIGH said "1-2 lines" —
  shipped at 13 code lines (5 of which are the
  explanatory comment block). The core is 5 lines:
  move name-read up, hash, setSeed.
- **Narrative AND topology now both kingdom-seeded.**
  The split 131 documented is closed. Every
  "Avalon" is the same place: same founder names,
  same nightmare-day, same constellation, **same
  island shape**.
- **Inline hash** avoids importing `_dreamHash` from
  story.js. story.js is story code; main.js doesn't
  need its internals. Two identical hashes living in
  different modules is acceptable at this scale.
  (If 3+ modules want this hash, extract to state.js
  utilities.)
- **Doesn't affect loadAndStart.** Saves restore their
  own seed via `save.js:111 setSeed(s.seed)`. 132's
  change only runs when a FRESH new game starts.
