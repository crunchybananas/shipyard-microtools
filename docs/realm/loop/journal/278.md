## why direct push (not chronicle())

The terminal entry CANNOT use `chronicle()`
because:
1. realm_fell.text writes via the
   normal `chronicle(text, beat.tag)`
   dispatch — that runs first.
2. realm_fell's `after:` runs AFTER the
   chronicle write.
3. `after:` sets `G.realmEnded = true`
   which activates the 260 chronicle
   gate.
4. Any chronicle() call AFTER step 3 is
   gated and no-ops.

So the terminal entry must EITHER fire
BEFORE step 3 in the same callback OR
bypass the gate with a direct push.

The clean approach: in the after-
callback, push the terminal entry
DIRECTLY to G.chronicle, then set
realmEnded. This:
- Keeps the chronicle gate's contract
  intact (no special "terminal" exception
  baked into chronicle()).
- Makes the terminal write atomic with
  the realm-end transition (one
  callback, one atomic action).
- Tagged 'requiem' for eviction-
  immunity per story.js:26
  `_EVICTION_IMMUNE_TAGS`.

## the prose

"The chronicle ends here. There is no
one left to begin a new page."

Two short sentences mirroring the
realm's existing register (declarative,
present-tense). The lift is the second:
"no one left to begin a new page" —
implicitly acknowledges:
- The chronicle has a BOOK-form (with
  pages)
- Pages are filled by people
- Without people, the next page stays
  blank

The CHRONICLE is the realm's primary
narrative surface; ending its book-
form is an act parallel to the realm's
last fire going out (the realm_fell
text just before).

The two requiem entries — realm_fell's
"last fire" + 278's "chronicle ends
here" — form a literal closure. The
realm DIES then the BOOK CLOSES.

## what I did NOT do

- **Add a separate NARRATIVE_BEATS
  entry for the terminal.** Considered
  briefly. Rejected because:
  (a) The terminal's trigger is
      identical to realm_fell's
      (`G.population === 0 && G.day > 1`)
      — duplicating it is messy.
  (b) Order in the array would matter:
      terminal BEFORE realm_fell would
      put it FIRST in the chronicle
      (wrong); terminal AFTER would
      have its chronicle write gated.
  (c) Putting the push in realm_fell's
      after-cb keeps the entire end
      transition in ONE place.
- **Modify `chronicle()` to allow a
  "terminal" exception.** Would
  complicate the gate's contract for
  one-time-only benefit. Direct push
  is more surgical.
- **Make the terminal prose dynamic
  (kingdom-name)**. Considered
  `"The chronicle of ${kingdomName}
  ends here..."` but that name-only-
  version reads more BOOK-formal than
  the realm's quiet observational
  register. Static prose matches the
  realm_fell beat's two-sentence
  rhythm.
- **Write a verify test asserting
  the chronicle gate IS active at the
  moment the terminal pushes.** Test
  #6 (existing) already verifies the
  chronicle gate; test #38 verifies
  the push happens via after-cb. The
  combination is sufficient.

## what about the rest of the chronicle being preserved?

Question worth considering: the
chronicle has 300 entries; the
realm-end transition writes TWO new
ones (realm_fell + terminal). When
the cap is reached, the eviction-
immune tags ('nightmare', 'stone',
'victory', 'requiem') survive
preferentially. Both new entries are
'requiem'-tagged → eviction-immune →
they will BE the last entries
preserved if 300+ entries existed
before realm-end.

Per 260 chronicle gate, no NEW
entries get added after realm-end,
so the cap won't trigger from the
realm-end transition itself.

The combination: the chronicle
ends EXACTLY at the realm's end
with two clearly-marked terminal
entries that will never be evicted.
Robust narrative-closure protocol.
