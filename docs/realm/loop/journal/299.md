## the build

`docs/realm/js/main.js` `_applyRealmEndFilter()`
— added lazy transition installation:

```js
let _realmEndTransitionInstalled = false;
function _applyRealmEndFilter() {
  if (G.realmEnded === _lastRealmEndedApplied) return;
  _lastRealmEndedApplied = G.realmEnded;
  if (!_realmEndTransitionInstalled) {
    _realmEndTransitionInstalled = true;
    canvas.style.transition = 'filter 1.5s ease';
    const post = document.getElementById('postfx');
    if (post) post.style.transition = 'filter 1.5s ease';
  }
  // ... existing filter assignment unchanged
}
```

~7 LoC added (5 transition + 2 boilerplate).

## why this matters

Pre-299: when realm fell, canvas filter
SNAPPED instantly from clear to grayscale-
brightness-dimmed. Visual acknowledgment of
the realm-end was abrupt — readable but
ceremony-less.

Post-299: filter FADES over 1.5s with
ease-out timing. The realm fades into
its grayscale state over the duration of
the requiem-beat-being-read. Cinematic
register matches narrative register.

The 1.5s timing is chosen for:
- **Long enough** for the realm-fall
  moment to feel ceremonial (1s feels
  rushed; 2s feels delayed)
- **Short enough** not to delay the
  player's visible acknowledgment that
  the realm has ended (>2s and the
  player wonders if something broke)
- **Matches existing 261 visual
  register**: not a flash, not
  obvious — a slow drain of color

## why lazy installation

The postfx canvas is created lazily by
`initPostFX()` (postfx.js:188). At the
time `_applyRealmEndFilter` is first
defined, the postfx canvas may not exist.

Installing the transition INSIDE the
helper, on first flip, ensures both
canvases exist when transition CSS is
applied. The `_realmEndTransitionInstalled`
flag prevents redundant installation
across subsequent flips.

Idempotent + correct.
