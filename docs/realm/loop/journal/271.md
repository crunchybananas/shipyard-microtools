## why this matters

Without the reset, a player whose realm
had named mayor/bard/smith/etc who clicks
"New Game" sees:
- The new realm's first tavern build
  calls `ensureMayor()` (per 034 hook,
  story.js:148) — which short-circuits
  because `G.namedCharacters.mayor`
  already exists from the previous realm.
- Same for bard / smith / merchant /
  teacher / rival — all 6 named-
  character ensure functions hit their
  "if already named, return" guard.
- Result: the new realm has the previous
  realm's mayor literally walking around,
  with the same name and same `appointedDay`
  (which is now in the FUTURE because
  the new realm's day=1).
- 270's inn_confluence_seen beat would
  fire IMMEDIATELY upon year3 reach
  because all 6 are already "named" in
  G state without ever having been
  introduced in chronicle.

This is silent state-leak across realm
boundaries. Same shape as 268's HIGH
finding (newGame doesn't reset
realmEnded), just for a different
field.
