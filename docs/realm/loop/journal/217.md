## the build

`docs/realm/js/render.js` — two changes.

### 1. Per-building sprite size table

`_SPRITE_SIZES` map added, one entry per
sprite-bearing building type. Sizes tuned by
inspection to match canvas drawX effective
footprints at zoom 1.0:

```js
const _SPRITE_SIZES = {
  granary: {w:36, h:42},  castle: {w:44, h:56},
  church: {w:40, h:56},   windmill: {w:38, h:52},
  tower: {w:32, h:52},    house: {w:36, h:40},
  tavern: {w:38, h:44},   blacksmith: {w:38, h:42},
  market: {w:42, h:38},   bakery: {w:36, h:42},
  barracks: {w:40, h:46},
};
```

`drawSpriteIfReady` now reads from this table;
draws bottom-anchored at `(s.x, s.y)`. Default
fallback `{w:36, h:42}` if a future sprite type
isn't in the table.

### 2. SVG path moved INSIDE scale/translate envelope

Critical structural change. Before 217:

```js
if (drawSpriteIfReady(...)) return;  // EARLY-RETURN
ctx.save(); ctx.translate(...); ctx.scale(1.1, 1.1); ...
switch (b.type) { canvas dispatch }
ctx.restore();
// damage cracks
// winter cap
```

Problem: the early-return SKIPPED the damage
cracks AND winter cap composition that come
AFTER `ctx.restore()`. SVG buildings would never
weather, never get snow caps.

After 217:

```js
ctx.save(); ctx.translate(...); ctx.scale(1.1, 1.1); ...
if (drawSpriteIfReady(...)) {
  ctx.restore();
} else {
  switch (b.type) { canvas dispatch }
  ctx.restore();
}
// damage cracks ← composes on either path
// winter cap ← composes on either path
```

SVG path now runs INSIDE the same 1.1× scale +
3px-y-offset envelope as canvas. After the
restore, downstream composition (damage,
winter cap, hover halo eventually) applies
uniformly regardless of which path drew the
building.

### Why no per-sprite tinting

The day/night tint is applied via
`applyNightTint()` as a screen-space multiply
blend AFTER the world pass (render.js line ~216).
Per-sprite tinting would DOUBLE-APPLY when
combined with the screen tint. Step 2 leaves
this alone — the screen tint already covers
both paths.

### verify-phaseb.mjs upgrade

Test now places ALL 11 sprite-bearing types in

* TH/2`), zooms 1.6× for sprite detail.
Screenshots saved at flag-off, flag-on, and
toggled-back-off.

## what I did NOT do

- **Per-sprite tinting.** Screen-space
  applyNightTint already covers both paths;
  per-sprite would double-apply.
- **Damage crack visual on SVG.** It composes
  structurally (paths merge after restore) but
  visually the crack is a black stroke pattern;
  on bright SVG fills it may read better than
  on canvas. Acceptable as-is.
- **Visual parity claim.** SVG at 36-44px is
  visually simpler than canvas hand-paint.
  Tuning to exact parity may not be achievable
  OR desirable — the SVGs ARE the new asset
  set; canvas is the legacy. Ship both, see.
- **Promote `_USE_SVG_SPRITES = true` default.**
  That's step 6. Steps 3-5 verify composition
  first.
