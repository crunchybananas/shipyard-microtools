## findings

### HIGH — newGame() does not reset G.realmEnded → new realm born already-dead

`docs/realm/js/main.js:301-324` defines
`window.newGame` (in-game restart) and
resets ~25 fields of G state (buildings,
citizens, particles, chronicle, storyFlags,
resources, day, season, stats, etc.). It
does NOT reset `G.realmEnded`.

**Consequence:** if a player whose realm
fell (G.realmEnded=true) clicks the
in-game "New Game" button, the new realm
starts with realmEnded still true. Effects:

1. **Chronicle gated** (story.js:40):
   beginGame() at main.js:155-168 calls
   `chronicle(foundingText, 'milestone')`
   to write the realm's first entry —
   this no-ops because realmEnded is true.
   The new realm has NO founding entry.

2. **Visuals desaturated** (main.js:585-
   600): `_applyRealmEndFilter()` sees
   realmEnded=true and applies
   `grayscale(0.85) brightness(0.85)` to
   the canvas. The new realm renders
   muted from frame 1.

3. **All NARRATIVE_BEATS gated**
   (story.js:1086): checkStoryBeats
   returns early. No chronicle entries
   ever fire for the new realm.

**Reproduced empirically** via `/tmp/
repro-268.mjs`:
```
FRESH GAME:    { realmEnded: undefined, chron: 1, filter: '' }
AFTER FALL:    { realmEnded: true,      filter: 'grayscale(0.85)…' }
AFTER newGame: { realmEnded: true, chron: 0, filter: 'grayscale(0.85)…' }
```

Same finding applies to `window.startNewGame`
at main.js:208-237 (no realmEnded reset
either). Title-screen flow back from a
fallen game state would also be broken,
but title-screen → newGame is normally
preceded by a page reload — so the
in-game restart path is the most-
exposed vector.

**Severity:** HIGH. The bug is silent
(no error thrown), reachable by the
expected "click New Game after realm
falls" flow, and breaks both the
chronicle (the realm's narrative
identity) AND the visuals (the realm's
visual identity) from frame 1.

**Recommended fix:** 1-line addition to
the newGame reset block (main.js:307-
324):

```js
G.realmEnded = false;
```

Plus the same in startNewGame for
defensive symmetry.

Verify test: assert that after
window.newGame() with G.realmEnded=true
pre-call, G.realmEnded is false post-
call AND the founding chronicle entry
appears.

### MEDIUM — newGame() does not reset sustained-state trackers

Same site (`main.js:307-324`). Does not
reset:
- `G.lastRaidDay` (211 sustained-peace
  tracker — economy.js:519)
- `G.lastDeathDay` (228 no-death
  tracker — economy.js:351 / combat.js:135
  / events.js:278 / soldiers.js:145)
- `G.lastUnderpopDay` (230 full-pop
  tracker — economy.js:333)

Stale values from the previous realm can
linger into the new realm. The numerical
math is mostly correct (G.day resets to
1, so `day - lastRaidDay` becomes
negative initially; trigger gates like
`>= 50` won't false-fire prematurely),
but it's fragile and not clean.

**Specific exposure**: if the previous
realm's lastUnderpopDay was 30, and the
new realm stays at full pop continuously
from day 1, the 230 full_pop_known beat
would fire at day 91 (30+60+1) instead
of the intended day 61 (1+60). Off by
30 days. Subtle but observable.

**Recommended fix**: reset these to
undefined alongside realmEnded.

### LOW — shooting-star particle (267) doesn't respect 400-particle cap

`docs/realm/js/story.js` 267 after-
callback unconditionally pushes one
particle. Neighboring particle spawn
sites in particles.js (lines 87, 91, 94,
115, etc.) gate on
`G.particles.length < 200/250/350/400`.

The 267 particle only spawns once per
realm (gated by once-per-realm flag), so
single-instance overflow is impossible.
But pattern-wise, the spawn-without-cap
breaks the convention. If 267 is the
template for future visual+textual
ticks (per the 267 [process] filing), a
2nd or 3rd application that spawns
multiple particles per fire could
trip the cap silently.

**Recommended fix**: add a single
`if (G.particles.length < 400)` guard.
≤2 LoC.
