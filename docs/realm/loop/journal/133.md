## what the returning player LOSES ACCESS TO

### tutorial

`ui.js:923-978`: tutorial is a single linear flow with
`tutorialStep` + `tutorialDismissed`. Neither state is
in G (confirmed by grep — the state lives module-local
in ui.js). Meaning:

- **Saved game → reload → tutorial state is FRESH
  (step 0).** Since 7-tick-old save dates that don't
  match current tutorial, tutorial state RESETS on
  reload.
- **But: tutorial auto-dismisses if** `G.gameTick > 30`
  (step 0's check) — so a returning player past day ~2
  bypasses tutorial entirely.

Net: **the migrant NEVER sees the tutorial on return.**
Tutorial is new-player-only. The tutorial is invisible
to returning players who already moved past step 0's
time-check.

**This is probably correct** — who wants to re-do
the tutorial? But it means the returning player has
zero guidance on what changed in the game since they
last played.

### keyboard shortcuts

18 mapped in 107. None of them are DISCOVERABLE from
within a loaded game. `?` / `/` opens a help overlay
(per input.js:327) but **only if the migrant remembers
to press one of those keys**.

A returning player who doesn't remember any shortcuts
can still play, but loses much of the UX design the
loop has shipped. The shortcuts are an expert-mode
layer; returning players default to mouse.

### the 127 welcome-back summary doesn't help migrants

127 fires on `visibilitychange` (tab blur → tab focus)
within the SAME session. It does NOT fire on load/
continue across sessions. The migrant opening the game
after a month away gets NO delta summary — just the
current state.

**FINDING — the migrant case is a separate UX slot
from 127's within-session summary.**

## HIGH finding — no "session delta" on Continue

Welcome-back (127) covers within-session absence; the
migrant case is cross-session. On Continue, the game
should ideally surface:
- Last save date (helps player situate temporally)
- "Since you last played: 0 days (the game was paused
  anyway)" — acknowledges the resume
- A MUTED nudge: "Your last chronicle entry was 'X'."
  so the migrant can re-connect emotionally.

Fix shape: on `loadAndStart`, after `loadGame()`, fire
a notify() with the last chronicle entry's text + day.
~10 lines in main.js.

```js
window.loadAndStart = () => {
  generateWorld();
  if (loadGame()) {
    // 133: welcome-back-cross-session. Show the last chronicle
    // entry so migrant reconnects to the realm's voice.
    const last = G.chronicle && G.chronicle[G.chronicle.length - 1];
    if (last) notify(`Where we left off (day ${last.day}): ${last.text}`, 'info', { chronicle: false });
    ...
  }
};
```

## MEDIUM finding — save date not stored

The save object doesn't record WHEN the save was
written (checked save.js). Adding `s.savedAt = Date.now()`
would let loadAndStart compute real-world days
between save and resume:
- "Your realm has been waiting for 3 days."
- "Your realm has been waiting for 2 months."

A subtle but meaningful re-engagement cue. The REALM
itself hasn't changed (it's paused while saved), but
the ACKNOWLEDGMENT that time passed in the player's
life makes the return feel significant.

Fix shape: save `savedAt: Date.now()` in save.js;
compute delta in loadAndStart; include in welcome-back
if > 1 day.

## LOW — no "what's new since your last save"

If the realm's loop shipped new systems between save
and resume (impossible with local-save + no
game-update, BUT relevant for a dev pushing new ticks),
there's no NOTES mechanism. Not practical for single-
player local game.

## LOW — chronicle filter state doesn't persist

078 shipped the chronicle filter (chips per tag). The
filter is module-local in story.js (`_chronicleFilter`
let). Reloading resets it. A player who set filter
to "stone" last session returns to "all." Minor.
