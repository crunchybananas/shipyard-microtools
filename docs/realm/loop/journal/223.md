Chrome online via Playwright headless.

## the build

`docs/realm/assets/sprites/windmill.svg` —
single `<animateTransform>` element wrapping
the 4 sail `<g>` blocks.

```svg
<g class="windmill-sails">
  <animateTransform attributeName="transform" type="rotate"
    from="0 64 71" to="360 64 71" dur="8s" repeatCount="indefinite"/>
  <!-- 4 sail groups (UP/DOWN/LEFT/RIGHT) -->
</g>
```

Hub center at (64, 71). 8s per full rotation
(~7.5 RPM) — slightly faster than realistic
(real windmills run 10-20 RPM in moderate wind)
but reads as "definitely turning" at iso scale
without becoming distracting.

The iron hub circles + bolts + highlight
remain OUTSIDE the rotation group so they
stay fixed (the bolts have positional
asymmetry that would visually rotate even
though they sit at the rotation center).

## verification

`verify-anim.mjs` (new, 70 LoC): loads sprite
sandbox, scrolls largest windmill into view,
takes screenshots at t=0 and t=4s (= 180° of
rotation), asserts frames differ.

Result: **frames differ** (30754B vs 30430B PNG
size; byte-identical false). Visual: sails
clearly in different positions across the
4-second gap. ✓

Animation runs in the SVG document's own loop
— no JS / no render.js change needed. Reaches
the live game automatically because Phase B
(default SVG sprites) shipped at 221.

## what I did NOT do

- **Touch render.js.** SVG-internal animation
  runs without JS involvement. The dispatch
  hook in `drawSpriteIfReady` continues to
  work as-is.
- **Animate the hub.** Bolts + highlight stay
  fixed because their visual asymmetry would
  rotate noticeably.
- **Vary rotation speed by wind / season.**
  Constant 8s. Could be parameterized later
  if the realm gains a `wind` state field.
- **Reverse direction occasionally.** Real
  windmills can change direction by yaw;
  this one is locked clockwise.
