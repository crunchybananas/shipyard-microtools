## meta-finding: gate-firing reliability under fastForward

The fastForward path (G.debug.fastForward
at main.js:535) batches 60 simTicks
per iteration with G.speed=60. This:
- IS reliable for: storyFlag-only gates
  (year2/year3/raidsSurvived/etc),
  building-presence gates, season-only
  gates (because seasons advance every
  7 days deterministically).
- ISN'T reliable for: dayPhase-fraction
  gates (current dayPhase = phase %
  dayLength × 60 + speed-adjusted; the
  60-tick batch jumps over precise
  phase windows), organic-lifecycle
  gates (citizens born require
  population > maxPop and happiness
  thresholds to spawn over multiple
  days, which the fast path can SKIP
  by batching).

**Implication**: verify-logic tests
that mock G.dayPhase + G.season +
G.storyFlags work for unit tests, but
[play] verification of dayPhase-
window beats requires real-time
gameplay (slow path). This is a
**verification limitation worth
documenting** for future [play]
ticks targeting specific beats.
