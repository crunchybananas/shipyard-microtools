## why this wasn't done sooner

The inline blocks survived because:

1. **034 added `after:` to the table late** — when
   034 introduced ensureSmith/Merchant/Teacher, it
   extended BUILDING_FIRST_BEATS with the new hook
   field, but didn't backport the inline
   tavern/barracks/church blocks (which pre-dated
   the table-with-after pattern).
2. **022 filled the TABLE from scratch** — the
   inline conditionals were from a prior era
   (pre-022). 022's refactor explicitly noted
   "decision ref 1" at the top of the file: the
   table is preferred for new beats.
3. **057 added to the table** without touching the
   inline blocks. 060's audit finally flagged the
   asymmetry.

Three ticks (022, 034, 057) could each have done
this; none did. 060 named it; 062 closed it. This
is the kind of tidy-up that happens when the
archivist does its job.

## what I did NOT do

- **Move `firstBirth` into the table.** It doesn't
  fit the `building-first-built` pattern — it's a
  stats-triggered beat (`G.stats.citizensBorn >= 1`),
  not `G.buildings.some(...)`. A different table
  (or a more general event-triggered beat system)
  would be needed. Out of scope for a cut-as-you-
  add tick.
- **Move `castleBuilt`/`firstRaid` into the table.**
  These have custom prose-composition logic
  (castle prose includes a named-character name +
  kingdom, raid prose is different tag). Could
  be generalized with a `text:` function instead
  of string, but also out of scope.
- **Move population threshold beats.** They're
  threshold-based, not first-built. Different
  system.
- **Touch the 060 HIGH echo-tag fix.** Still teed
  up. One focused change per tick.
- **Add entries for new beats** — this is a
  consolidation tick, not a content tick.
