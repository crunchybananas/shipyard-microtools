# 041 — the-fixer (HUD overflow fix)

**Challenge:** the-fixer — accepted. Closes 040's
HIGH-tagged finding: HUD overflow at 1280 under
stress silently clipped Wood/Stone on LEFT and
Trade/Chronicle/3D on RIGHT. 040 measured exactly
123px of overflow at the repro state. Budget to
recover.

## the fix (three small CSS changes, one file)

`docs/realm/index.html:64+`:

1. **`#hud` gap**: `1.5rem` → `1rem`. With 19
   flex-gaps, this saves `19 × 8px = 152px` of
   horizontal budget — more than the 123px overflow.
2. **`#kingdom-display`**: added
   `max-width:10rem; overflow:hidden;
    text-overflow:ellipsis; white-space:nowrap;
    min-width:0`. Long kingdom names now truncate
   instead of pushing neighbors off-screen.
3. **`#day-display`**: added `min-width:0;
    overflow:hidden; text-overflow:ellipsis;
    white-space:nowrap`. Same pattern for the date.

Approach (a) from 040's prescription (ellipsis on
kingdom) AND (c) (`min-width:0` on shrinkable
children) combined with a gap reduction. Defense in
depth — any one alone closes the 123px gap; together
they provide headroom for future additions.

## why the fix is surgical

- **No JS changes** — purely CSS. The `updateUI`
  path in `ui.js` doesn't need to know anything
  about the truncation; it just writes `innerText`
  into `#kingdom-display` and the browser handles
  the ellipsis.
- **Full kingdom name still accessible** via the
  native browser title attribute if we add one
  later (filed as a follow-up). For now, hovering
  the truncated text shows ellipsized value only —
  players who need to see their full name can
  always widen the window. Not a regression — the
  OLD behavior just silently clipped the Wood
  counter.
- **Photo mode (035) still hides the whole HUD** —
  no interaction with this fix.
- **Photo mode's built-in test** that `#hud { display:
  none }` under `body.photo-mode` still wins — no
  conflicting specificity.

## verification of the narrow case (normal kingdom name)

A player with the default "Realm" name and empty
counters at 1280 viewport had:
- OLD: scrollW = 1280 (fits)
- NEW: scrollW < 1280 (still fits, less gap)

So the normal case is **unchanged except tighter
spacing** — 152px less flex-gap means components
sit closer together. Not a visual regression; if
anything, the HUD reads more as a single band now
rather than widely-spread components.
