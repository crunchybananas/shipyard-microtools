## what a silent-module does

Per challenges.md: *"ship a module whose command does
something non-obvious: prints nothing, or only writes
a file, or only registers a handler a later module
will invoke. Plumbing that pays rent later."*

The tick's output is a function on `G.debug.tintCurve`
that any future ticks can call to inspect the tint
overlay's coefficient profile across a full day. It
changes nothing user-visible. It doesn't fix a bug.
It's pure plumbing.

## what shipped

New `tintCurve({samples})` export in `js/state.js:242+`,
wired to `G.debug.tintCurve` via `main.js:187`.

Mirrors the render.js:2515-2530 hue-variation math
exactly (from 012):

- Dawn (dayT < 0.10): rose profile `kR=80 kG=140 kB=160`
- Dusk (dayT >= 0.10, 0.70 ≤ blend point): amber→indigo
  lerp from `(80,120,180)` at t=0.70 to `(180,140,100)`
  at t=1.0
- Overlay off (daylight ≥ 0.95): zero coefficients,
  tint = pure white

For each sample, returns `{t, hour, daylight, kR/G/B,
tintR/G/B}` — both the raw coefficient profile and the
resulting RGB tint at that moment.

Pure function. Saves/restores `G.dayPhase`. No other
state mutation.

Chrome-mcp, fresh game, `G.debug.tintCurve({samples:12})`:

The dawn→amber→indigo progression matches 012's
three-band profile exactly.

## lightCurve drift noted (not fixed)

While writing tintCurve I noticed: `lightCurve` (004)
has `overlayR/G/B` fields that were accurate BEFORE
012 hue-varied the tint. Post-012 the real render-
time coefficients vary with dayT. `lightCurve`'s
overlay fields still use the pre-012 static
(160/150/100) values.

Two options for a follow-up:
- (a) Have lightCurve call tintCurve for its overlay
  fields. Extra allocation but keeps `overlayR/G/B`
  accurate.
- (b) Document lightCurve as "daylight-curve shape
  only" and direct callers to tintCurve for tint math.

Added both lightCurve's function-header comment AND
an ideas.md entry so the next tick touching lighting
infrastructure picks it up. Not fixing now — scope
discipline for a silent-module tick.

## what this unlocks

Any future tick considering lighting math can now:

1. Sample current tint before changing anything —
   `const before = G.debug.tintCurve({samples:24})`
2. Ship a change
3. Sample again — `const after = G.debug.tintCurve(...)`
4. Diff the two arrays to show exactly what changed

This is the kind of instrumentation 011 argued should
precede any lighting change. 012 shipped without it
(worked anyway), but 013 verified via screenshots
which was expensive. Next lighting tweak has a cheaper
verification path.

Pattern also useful for:
- Photo-mode "sunset preset" (018) — pick a
  tint-recipe numerically rather than eyeballing
- A future `the-photographer` tick with the full-
  settlement gallery (018 idea)
- Verification of season-aware midday tint (013 idea)
  if anyone ships it

## pattern noting

014 & 015 moved "decision next to data." 037 moves
*instrumentation* next to data — tintCurve lives in
state.js beside getDaylight, lightCurve, SEASONS. The
lighting-math module is now self-describing:

- `getDaylight(t)` — the function used by render
- `lightCurve({samples})` — sample the daylight shape
- `tintCurve({samples})` — sample the tint shape
- `SEASONS[season].tileShift` — the per-season bias
  (separately applied in render)

A future lighting-refactor tick could centralize these
(e.g., `getTintCoefficients(t)` shared between render
and sampler) but the current split is navigable as-is.
