## the build

`docs/realm/js/story.js` — single-line addition
to the `realm_fell` NARRATIVE_BEATS entry, using
the **`after:` callback schema** (144):

```js
{ flag: 'realm_fell', tag: 'requiem', onFire: 'requiem',
  trigger: G => G.day > 1 && G.population === 0,
  text: G => { /* unchanged */ },
  after: G => { G.realmEnded = true; } },
```

When the realm_fell beat fires (population
drops to zero post-day-1), `G.realmEnded`
becomes true. Future rendering / audio / UI
ticks can branch on this flag.

8 LoC including 6-line comment explaining
the silent-module intent.

## why this shape

- **`after:` callback over imperative.** 144
  added the `after: (G) => void` schema field
  for arbitrary side effects on NARRATIVE_BEATS
  entries. realm_fell is a perfect fit: the
  beat ALREADY fires on the right condition;
  adding a flag-set is the single clean side
  effect. No new dispatch path needed.
- **Second use of `after:` schema.** 144's
  namesake citizen-rename was the first.
  192 is the second. 188 invariant rule:
  "new schema fields are added when 3+
  entries would need them" — `after:` had 1
  use until now; 192 makes 2. Still under
  the 3+ threshold for promoting new fields,
  but trending toward justified.
- **Silent-module shape.** No immediate
  visual / audio / UI change. The flag exists;
  consumers don't yet. Future ticks shape
  the consequences:
  - Render desaturation post-end
  - Fog return
  - UI silence (no toasts post-end)
  - Audio fade or silence
  - "Game Over" visual treatment
- **Closes the 103 [code] thread cleanly.**
  103's filed idea: "G.realmEnded flag: set
  when requiem fires. Enables post-end
  rendering (fog return, desaturation,
  silence). Future mood tick." Shipped
  exactly to spec.
- **No risk.** The flag has no consumers. If
  it's never read, it has zero side effect.
  If a future tick reads it, that tick
  decides what behavior to add.
