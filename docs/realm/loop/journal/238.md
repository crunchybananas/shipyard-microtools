## the methodology

`scripts/verify-critic.mjs` (new, 100 LoC):
- Boot live game via Playwright + auto-server
- Place 12 buildings (one of each sprite-type
  + 2 houses) in a clustered settlement at
  tiles (36-50, 36-39)
- Center camera on cluster
- Capture 7 screenshots:
  - 4 lighting variants (dawn/midday/dusk/night
    via `G.dayPhase` injection)
  - 1 close zoom (2.2×)
  - 1 far zoom (0.6×)
  - 1 winter season

Zero page errors across all captures. Sprite
cache 22/22 ready (11 default + 11 per-realm
variants for the test kingdom).

## findings

### HIGH — SVG sprites read smaller than canvas at default zoom

At zoom 1.4 (default), the 12-building cluster
appears as small blobs. Individual sprite
identity is hard to distinguish. The canvas
drawX functions had hand-painted detail that
gave each building a distinctive silhouette
even at distance; SVGs at 36-44px lose that
distinctness.

This confirms 217's filing: "_SPRITE_SIZES
table tuned for zoom 1.0; at zoom 1.5+ SVG
sprites may look small relative to canvas
drawings." After 220 ticks of empirical data,
the sizing is genuinely undersized for the
actual default-zoom (which is 1.3 per
state.js, not 1.0).

**Recommendation:** bump `_SPRITE_SIZES` by
~20-30%. Test values:
```js
granary {w:42, h:48}    (was 36×42, +17%)
castle  {w:54, h:68}    (was 44×56, +21%)
church  {w:48, h:68}    (was 40×56, +20%)
windmill{w:44, h:64}    (was 38×52, +16/+23%)
tower   {w:38, h:64}    (was 32×52, +19/+23%)
house   {w:42, h:48}    (was 36×40, +17/+20%)
tavern  {w:46, h:54}    (was 38×44, +21/+23%)
blacksmith {w:46, h:52}  (was 38×42, +21/+24%)
market  {w:50, h:46}    (was 42×38, +19/+21%)
bakery  {w:42, h:50}    (was 36×42, +17/+19%)
barracks{w:48, h:56}    (was 40×46, +20/+22%)
```

~20% larger across the board. Verify via
verify-phaseb screenshots.

### MEDIUM — silhouettes blur in dense placement

When sprites are placed adjacent
(cx+=3 spacing), individual identities merge
visually. The Phase A SVGs are designed for
free-standing display in the sandbox; the
live game places buildings shoulder-to-shoulder
in dense settlements.

**Recommendation:** Defer for now. The 20%
size bump above will help. If issues persist,
consider adding sprite-edge stroke
(building-outline) on hover OR distinct
per-building backdrop tint.

### LOW — winter caps + SVG sprites compose but visually mixed

The 218-verified winter cap composition still
works on the SVG path (verified: caps render
on SVG buildings). But the contrast between
simpler SVG silhouettes + canvas-style
`_WINTER_CAPS` ellipse-overlays creates a
slightly inconsistent visual register —
elaborate caps on simple silhouettes.

**Recommendation:** Defer. 218 filing already
noted potential "per-path cap tuning" if
visual review flags this — flagged here, but
not urgent. The look is functional.

### LOW — animations require live-frame inspection (not single screenshots)

The 10 active SVG animations (Phase C+D) all
verify via verify-anim's t=0/t=N PNG diff.
Single audit screenshots can't capture motion,
so this critic tick can't directly evaluate
animation quality.

**Recommendation:** A separate animation-quality
review tick using HEADED Playwright + manual
observation, OR a multi-frame GIF capture
per sprite. Filed.

### POSITIVE — day/night tinting composes cleanly across paths

Dawn → midday → dusk → night screenshots all
show the `applyNightTint` screen-multiply blend
working uniformly. No over-tinting, no
under-tinting, no sprite-vs-canvas inconsistency
in coloration. The Phase B step 2 decision to
defer per-sprite tinting (let the screen-space
tint cover both paths) was correct.

### POSITIVE — zero page errors, sprite cache healthy

22/22 cache entries ready. No console errors
across all 7 screenshots in 4 lighting + 2
zoom + winter scenarios. Phase B+C+D
infrastructure is genuinely stable.

### POSITIVE — variant pipeline works invisibly

The test realm got its kingdom-hashed church
glass color (and other variants when their
sprites have palettes); no errors, no flicker.
The 220 string-replace pipeline is solid.

## what this finds about the loop

**The SVG arc shipped, but sprite SIZING is the
dominant remaining visual concern.** All 75
ticks of effort produced functional integration,
animation, and variants — but the user-facing
pixel size of individual buildings reads as
small at default zoom. This is a 1-line tuning
fix waiting for the value table.

The live game looks DIFFERENT from before
(simpler silhouettes; SVG aesthetics; Phase D
animation polish), not necessarily WORSE — it's
the new asset set. But the sizing should be
visually neutral at minimum.

## what I did NOT do

- **Apply the size bump.** That's a fixer tick
  (239). 238 is review-only; recommendations
  filed.
- **Compare canvas vs SVG side-by-side.** The
  toggle exists (`window.__realm.toggleSVG`)
  but the audit was about CURRENT visual state,
  not regression. Defer to dedicated comparison
  tick if needed.
- **Audit sprite-internal quality.** The 11
  SVG files are individually reviewable in
  the sandbox; this audit was about SYSTEM-
  level visual integration.
- **Multi-zoom systematic test.** Only 3 zoom
  values (0.6/1.4/2.2). A finer sweep
  (0.5/0.75/1.0/1.25/1.5/2.0/2.5) would
  produce cleaner sizing data. Filed.
