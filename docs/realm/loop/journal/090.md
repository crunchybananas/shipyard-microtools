## the change

`docs/realm/js/story.js` — new `NARRATIVE_BEATS` table +
single dispatch loop, replacing 4 distinct inline shapes:

- `firstBirth` — was a 4-line `if` block
- `popCheck` — was a 5-entry 2D array + 5-line loop
- `castleBuilt` — was a 5-line `if` with dynamic text inline
- `firstRaidSurvived` — was a 4-line `if` block

All 8 beats (firstBirth + 5 pop + castle + raid) now live in
the NARRATIVE_BEATS table. Each entry: `{ flag, tag, trigger(G)
→bool, text(string | (G)→string) }`. Dynamic text supported
for castleBuilt's mayor-or-heralds branch.

Dispatch loop in `checkStoryBeats`:

```js
for (const beat of NARRATIVE_BEATS) {
  if (!hasFlag(beat.flag) && beat.trigger(G)) {
    setFlag(beat.flag);
    const text = typeof beat.text === 'function' ? beat.text(G) : beat.text;
    chronicle(text, beat.tag);
  }
}
```

## why this shape

- **Mirrors BUILDING_FIRST_BEATS** (062 cut-as-you-add shipped
  that pattern). Now state-triggered beats have the same
  table-driven shape as building-triggered ones.
- **Trigger as function, not as predicate lookup.** 062's
  original spec said `trigger: (G) => bool`. Keeping it a
  function (vs e.g. `{ type: 'stat', key: 'citizensBorn',
  threshold: 1 }`) means future entries can express arbitrary
  conditions without schema churn.
- **Text as string OR function.** String for the 7 static
  cases; function for castleBuilt's mayor-or-heralds branch.
  Future entries (e.g. "a kingdom-named beat" or "references
  named character if present") can use the function form
  without the table growing new fields.
- **Happiness beats stay inline** (story.js:320-340). They
  have hysteresis reset semantics — when happiness falls back
  into the [35,65] band, two flags clear so the beat can
  re-fire. That's fundamentally different from the one-shot
  trigger the table encodes. Forcing hysteresis into the table
  would add special-case fields for 1 entry.
- **First-snow (088) stays inline** for the same reason —
  it's a simpler story-specific beat tied to 072 founders +
  season transition, not a generic state threshold.

## what I did NOT do

- **Migrate happiness peak/crisis.** Would require adding
  re-fire semantics + flag-reset fields. Big scope creep;
  they're fine inline where their uniqueness is visible.
- **Migrate first-snow (088).** Trivial to add but would
  creep toward "migrate everything" which is a different
  tick.
- **Migrate enhancements.js year-milestone beats** (073
  filed). Separate tick; filing as follow-up.
- **Add new beats.** Refactor-only constraint — zero new
  user-visible behavior. This tick only restructures.
- **Hoist dispatch loop out of checkStoryBeats.** Would be
  gratuitous — the loop is 5 lines, reads fine inline.
- **Invert the order of BUILDING_FIRST_BEATS and
  NARRATIVE_BEATS checks.** Same execution order preserved
  (building beats fire before state beats within the tick).
