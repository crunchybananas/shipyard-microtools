## the 016 #1 re-diagnosis (why it got dropped)

016 proposed capping `G.raidFlash` accumulation because
"4 rapid raids left it stuck at 0.35 long after ended."
The hypothesized fix was `Math.min(G.raidFlash + delta, 1.0)`.

Reading the code:

- `economy.js:445`: `G.raidFlash = 1.0` — **direct
  assignment**, not `+=`. No stacking, ever.
- `render.js:2666`: `G.raidFlash = Math.max(0, G.raidFlash - 0.018)`
  — decay per render frame.

So the 016 stacking hypothesis was wrong. What 016
actually saw was the flash decay pausing when chrome-mcp
backgrounded the tab. The decay lives in `render()`; when
the tab is hidden, rAF pauses, and `raidFlash` freezes at
whatever value it had last rendered. In normal play
(visible tab, 60fps) the decay takes about 55 frames ≈
0.9s after a raid — correct behavior.

Dropped the idea in ideas.md with a full explanation. The
decay-tied-to-render-frame issue IS real for testing
environments, but fixing it means moving decay into the
game-tick loop in main.js — more than a `[code]` fix warrants when no real-player bug exists. Filed as a
follow-up note inside the dropped-item rather than a new
active idea.

## the actual fix

007 Finding #2: *"tutorial progress label `2/10` is
ambiguous — reads like 2/10 resources, not step 2 of 10."*

Site: `js/ui.js:937`:

```diff
- <div class="tut-progress">${tutorialStep + 1} / ${TUTORIAL_STEPS.length}</div>
+ <div class="tut-progress">Step ${tutorialStep + 1} of ${TUTORIAL_STEPS.length}</div>
```

One string. One render path. That's the whole fix.

## verification

Chrome-mcp, fresh New Game:

```js
document.querySelector('.tut-progress')?.textContent
→ "Step 2 of 10"
```

Was previously `"2 / 10"`. Unambiguous now — reads as
tutorial progress, not a resource stat.

`node --check js/ui.js` → OK.
