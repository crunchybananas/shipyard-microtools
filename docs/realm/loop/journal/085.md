# 085 — the-fixer (eviction-immune once-per-realm beats)

**Challenge:** `the-fixer` — accepted. Ships 083's
filed defensive idea: enforce 084's noise-evicts-
signal invariant at the code level. Once-per-realm
beats (nightmare / stone / victory) now survive
the 300-entry chronicle cap even under heavy
concurrent writes.

## the invariant we're defending

084 codified this in narrative-surfaces.md:

> "If any system writes >1 chronicle entry/game-day
> in steady state, it can push rare one-shot beats
> out of the 300-entry window before they're read."

082 saw this in the wild — 197 raid-toast entries
evicted nightmare/stone/echo/dream before the audit
could sample them. 083 fixed THAT instance by
suppressing the raid-toast writers. 085 makes the
chronicle system itself ROBUST to future
regressions of the same shape.

## the fix

`docs/realm/js/story.js:17-44` (replaced 1-line cap
with guarded logic):

```js
const _EVICTION_IMMUNE_TAGS = new Set(['nightmare', 'stone', 'victory']);

export function chronicle(text, tag='misc') {
  initChronicle();
  G.chronicle.push({ day: G.day, season: G.season, tick: G.gameTick, text, tag });
  // Cap to last 300 entries, preserving eviction-immune tags
  if (G.chronicle.length > 300) {
    const excess = G.chronicle.length - 300;
    const toRemove = [];
    // Collect oldest-first indices of non-immune entries, up to `excess`
    for (let i = 0; i < G.chronicle.length && toRemove.length < excess; i++) {
      if (!_EVICTION_IMMUNE_TAGS.has(G.chronicle[i].tag)) {
        toRemove.push(i);
      }
    }
    // Remove in reverse so earlier indices stay valid
    for (let j = toRemove.length - 1; j >= 0; j--) {
      G.chronicle.splice(toRemove[j], 1);
    }
  }
}
```

Three immune tags chosen:

- **`nightmare`** — once per realm, 043. The
  rarest beat in the game.
- **`stone`** — once per realm, 056. Includes both
  the discovery and 079's offering (same tag).
- **`victory`** — castleBuilt beat + enhancements.js:
  5465 victory v2. The end-of-act moment.

Not immune (deliberately):
- **`milestone`** — first-build beats, pop
  thresholds, happiness thresholds, etc. Many
  entries per realm. Some ARE one-shot but
  collectively take up too much space to preserve
  all.
- **`character`** — 034 ensures + 072 founders +
  bard/mayor beats. Notable but recurring in some
  systems.
- **`birth`/`death`** — tied to real events that
  recur.
- **`raid`/`echo`/`season`/`dream`/`event`/
  `research`/`misc`** — all recurring or numerous.

The three immune tags are the ones where losing a
single entry is PERMANENT data loss. Each is
designed to fire exactly once per realm; without
it, the realm has no record of that moment ever.

## soft-overflow behavior

If the entire chronicle is eviction-immune (305
victory entries stacked), the cap can't reduce
the buffer. Current behavior: soft-overflow to
306, 307, etc. Acceptable because:

1. In practice impossible — only 3 immune tags,
   each once-per-realm. Max 3 immune entries per
   realm. Can never exceed 300 via immune alone.
2. If a future change floods an immune tag with
   noise (e.g., adds 100 `victory` writes), the
   overflow flags the bug rather than hiding it.

Budgeted for this. Test 4 below confirms.

## what this enables

- **Future noise regressions** (e.g., a new system
  that writes more than 1 entry/day without
  `{chronicle: false}` on UI toasts) no longer
  silently delete nightmare/stone/victory. The
  regression shows up as a bloated chronicle
  instead — visible, auditable.
- **200+ day realms** (fastForward audits) can now
  reliably sample once-per-realm beats at end
  state. 082's failure mode won't recur.
- **Future tag additions** (if narrative grows)
  can opt into immunity by adding to the Set.
  One line.
- **CI/audit scripts** can assert "a post-200-day
  realm contains the kingdom's seeded nightmare/
  stone entries." Previously this was flaky under
  noise; now deterministic.

## what this does NOT change

- **Normal chronicle display**: Players still see
  entries in time order. Immunity is a cap-policy
  detail, invisible during rendering.
- **059 echoes**: source candidates still come
  from the same list. Immunity doesn't affect
  echo eligibility.
- **078 tag-filter UI**: filters work as before.
  Immune entries filter the same as anything else.
- **Non-immune tag behavior**: milestone/character/
  dream/etc still evict oldest-first as before.

## what I did NOT do

- **Add more immune tags.** Considered `research`
  (some tech beats are one-shot) and `birth` (first
  birth is firstBirth flag). But:
  - Research has 11+ entries per realm (per-tech),
    not all one-shot.
  - `birth` tag fires on every new citizen via
    economy.js:362, many per realm.
  Both would bloat the immune set. Kept to 3
  surgical entries.
- **Flag-based immunity** (e.g., mark individual
  entries eviction-immune via a `protect: true`
  field on the entry). Would be more flexible but
  adds per-entry storage and requires writers to
  opt in. Tag-based is simpler for this pass.
- **Console warning when soft-overflow engages.**
  Budgeted for, not needed now. If a future system
  regression floods immune entries, the overflow
  would be discovered by chronicle audits. File-
  level for later.
- **Update narrative-surfaces.md.** 085 enforces
  the invariant 084 codified; the doc already
  documents the policy. This is the
  IMPLEMENTATION of what was already written.
  075 rule: "update doc in same commit" applies
  when SEMANTICS change; here only the
  enforcement mechanism.
