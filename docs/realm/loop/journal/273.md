### Change A — lightning particle render

`docs/realm/js/render.js` — added
`'lightning'` render branch after
'dustmote', before 'shootingstar':

```js
} else if (p.type === 'lightning') {
  const baseX = s.x + (p.offsetX || 0);
  const baseY = s.y + p.offsetY;
  const flicker = (p.alpha < 0.4 && p.alpha > 0.2) ? 0.5 : 1.0;
  // Big soft halo (warm-white, contrasts with night-blue sky)
  ctx.globalAlpha = p.alpha * flicker;
  const halo = ctx.createRadialGradient(...);
  halo.addColorStop(0, 'rgba(255, 250, 220, 1.0)');
  halo.addColorStop(0.4, 'rgba(255, 240, 180, 0.5)');
  halo.addColorStop(1, 'rgba(200, 180, 120, 0)');
  ctx.fillStyle = halo;
  ctx.beginPath();
  ctx.arc(baseX, baseY, 70, 0, Math.PI * 2);
  ctx.fill();
  // Bright core
  ctx.fillStyle = 'rgba(255, 255, 255, 1.0)';
  ctx.beginPath();
  ctx.arc(baseX, baseY, 5, 0, Math.PI * 2);
  ctx.fill();
}
```

Two-phase brightness: bright initial
flash, then a flicker phase between
alpha 0.2 and 0.4 (50% reduction)
mirrors real distant lightning's
primary-then-secondary brightness.

~28 LoC + comment.

### Change C — 267 spawn-math correction

While testing 273's lightning, found
that BOTH 267 (shooting-star) and 273
(lightning) were using `cx =
G.camera.x / 32, cy = G.camera.y / 16`
as tile coordinates — but THIS IS NOT
A VALID TILE-COORD CONVERSION. The
actual inverse iso projection is:

```js
const tx = (G.camera.y / 16 - G.camera.x / 32) / 2;
const ty = (G.camera.y / 16 + G.camera.x / 32) / 2;
```

For default camera (0, 1280): old math
gave (0, 80) → toScreen returns
(-2560, 1280) → particle BASE position
far off-screen → render-cull at
render.js:2584 might trigger.

For default camera (0, 1280): new math
gives (40, 40) → toScreen returns
(0, 1280) → particle BASE position
exactly at camera focus → render
proceeds reliably.

The 267 shooting-star happened to be
visible in earlier tests because
camera was set to (256, 1344) for the
townhall A/B, which by coincidence
positioned the star in viewport.
With default camera, both particles
would have been culled.

**Fixed both 267 and 273 spawn paths**
with the corrected inverse iso math.
This is a latent-bug fix that 268
pessimist didn't surface (the bug was
masked by my manual camera positioning
in earlier visual tests).

~10 LoC of correction across both
after-callbacks + comment.

## why warm-white-yellow halo, not pale-blue

Initial implementation used pale-blue
colors (matching "sky-lit cloud").
First Playwright A/B at default zoom +
night sky showed the pale-blue halo
was almost invisible against the
night-tinted blue sky background. The
postfx blue-tint pipeline (postfx.js
lines 89-91 — winter-style cool-blue
palette) was COLLIDING with the
particle palette — same hue family,
no contrast.

Switched to warm-white-yellow
(255,250,220 → 255,240,180) which
matches how real distant lightning
ILLUMINATES rather than IS — clouds
struck by a bright bolt look pale-
yellow-white, not pale-blue. The
warm hue contrasts strongly against
night-blue sky. Visually-coherent
AND legible.

Lesson: when pixel-shipping particles
into a sky background, choose a
color OFF-AXIS from the sky's
dominant hue family. Document as
follow-on observation.
