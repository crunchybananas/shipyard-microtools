## the bug

010 (the-gamer, Hard + Military) observed: the raid-incoming
toast reads *"No defenders! Build walls, barracks, or towers."*
— but on a fresh game, none of those three appear in the
build bar. They're all tech-gated. The toast points at
invisible solutions. A Military-scenario novice hits a wall.

## the fix

Site: `js/economy.js:383` — the `else` branch of the
defenders-check inside `checkRaids()` when the raid fires.

Imported `isBuildingUnlocked` from `tech.js` (already written
and exported — sibling to the existing call sites in ui.js).
Replaced the single static hint with a four-branch
conditional based on which defensive buildings are actually
unlocked:

- `!wallOK && !barrackOK`: *"Research Masonry to unlock walls."*
- `wallOK && !barrackOK`: *"Build walls. Research Military for barracks/towers."*
- `!wallOK && barrackOK`: *"Build barracks or towers."* (prereq chain makes this unreachable in practice but handled defensively)
- both: *"Build walls, barracks, or towers."* (original text preserved)

Both `wall` and `barracks` cover the relevant gating. Towers
share `barracks`'s tech (both under `military`), so one check
suffices for both.

Code is additive. No call-site changes elsewhere; the toast
still fires through the same `notify(line, 'danger')` pipeline.

`node --check js/economy.js` passes.

## verification

Chrome-mcp, fresh New Game (defaults Peaceful Easy). Ran all
three tech states through the raid-fire path:

| State                     | Toast produced                                                          |
| ------------------------- | ----------------------------------------------------------------------- |
| no military techs         | ⚠️ No defenders! Research Masonry to unlock walls.                       |
| +masonry                  | ⚠️ No defenders! Build walls. Research Military for barracks/towers.    |
| +metallurgy+military      | ⚠️ No defenders! Build walls, barracks, or towers.                       |

The novice path (no techs → masonry → masonry + military) now
reads as a staircase instead of a single confusing line. Each
step names exactly what's available plus the next research
target.

Also noticed during testing: the initial phrase I shipped
used a naive `.join(' or ')` which produced *"walls or
barracks or towers"* for the fully-unlocked case — three
"or"s, awkward. Fixed during the same tick by special-casing
the fully-unlocked branch to preserve the original Oxford-
comma phrasing.

## what I noticed

1. **`isBuildingUnlocked` was already the right tool.** The
   function existed in tech.js with a clean signature (`id ->
   boolean`, accounts for `FREE_BUILDINGS`, iterates techs).
   No new infrastructure needed. This is the kind of fix
   that's only 8 lines because someone before wrote good
   helpers.

2. **The toast is fired from `checkRaids()`, not
   `notifications.js`.** I briefly considered moving the
   rephrase logic into `notify()` itself (detecting building
   mentions in text, rewriting). That's tempting but wrong:
   it couples a generic channel to a specific content
   concern, and the "smart" behavior varies by content.
   Better to keep the logic at the callsite.

3. **This is a pattern.** 014 moved the `raidsSurvived`
   increment into the same function block where raid state
   actually changes. 015 does the analogous move for the
   hint — put the decision next to the data that informs it.
   Both fixes reduce side-effects-at-a-distance; the common
   theme is "information should live where it's used."
