## the build

Two small changes in `docs/realm/js/story.js`.

### 1. Dispatch loop extended with `after` hook

After the `onFire` audio call in `checkStoryBeats`:

```js
if (typeof beat.after === 'function') {
  try { beat.after(G); } catch (_e) {}
}
```

Schema is now `{ flag, tag, trigger, text, onFire?,
after? }`. Backward-compat: entries without `after`
fire no side effect. Chronicle is the contract;
side effects are garnish.

### 2. 134 namesake entry gets `after`

```js
after: G => {
  if (!G.citizens || G.citizens.length === 0) return;
  const kname = G.kingdomName || 'Realm';
  const idx = _dreamHash(`${kname}_namesake_who`) % 3;
  const key = ['founder1', 'founder2', 'founder3'][idx];
  const f = G.storyFlags[key];
  if (!f) return;
  const c = G.citizens[G.citizens.length - 1];
  if (c && c.name !== f) c.name = f;
}
```

Recomputes the same founder pick as `text()`, then
renames the NEWEST citizen (last in `G.citizens`)
to that founder's name. Citizens are appended to
the array on birth (confirmed via `makeCitizen` in
world.js), so the newest is the right
"just-born-child."

## why this shape

- **Hook at the entry level, not at text-composition.**
  Keeping `text()` pure (just returns a string)
  preserves the dispatch-loop contract. `after()` is
  explicit side-effect territory.
- **Same founder-pick math duplicated** — text()
  and after() both independently compute the name.
  Not DRY, but avoids coupling the after-hook to
  text()'s return value. A future refactor could
  extract `pickNamesakeFounder(G)` as a helper.
- **try/catch around `after()`** — defensive.
  Chronicle is already written by this point; if
  the after-hook throws, the beat still fires
  narratively. Chronicle > side effects.
- **Mirrors BUILDING_FIRST_BEATS `after:` field**
  shipped at tick 034. Consistent schema across
  both tables.
- **`c.name !== f` guard** — idempotent. If
  someone calls this twice (shouldn't happen given
  the `hasFlag(beat.flag)` gate, but defensive),
  we don't no-op re-rename.

## what this changes

**Before 144:** 134 fires a chronicle beat; no
citizen in the game actually carries the name.
Text references it but it's theatrical.

**After 144:** 134 fires the beat AND a real
citizen (the newest one) gets renamed. If the
player clicks that citizen, their tooltip shows
"Lira" (or whichever founder). Makes the beat
material.

**Persistence:** citizen names are saved in
`save.js` (verified: `name:c.name` at save.js:29).
Reload preserves the renamed citizen.
