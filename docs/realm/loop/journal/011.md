# 011 — the-contrarian (against 005)

**Challenge:** the-contrarian — accepted. 005 strongly
recommended re-shaping the `lightCurve` with peaks at
t=0.22/0.78, a dropped night floor, and sky/ground
coupling. It's been sitting un-touched for 5 ticks.
Somebody should try to knock it down before anyone codes
it.

This review does not argue 005 is wrong on everything —
just that the proposed shape is wrong, the diagnosis is
partly mis-aimed, and a cheaper path exists.

## what 005 got right (stated first, honestly)

- **Observation**: the curve is flat-plateau with cliffs,
  not an arc. True. 004's numbers prove it.
- **Observation**: sky gradient peaks warm at t=0.22/0.78
  while ground luminance is either still ramping or already
  descending at those moments. True.
- **Observation**: the multiply-blue tint overlay is
  perceptually dominant. True — this is the key observation,
  and 005 named it even while proposing the wrong fix.

Three real observations. I don't dispute any.

## what 005 got wrong

### 1. The proposed fix violates 005's own constraints

005's "what NOT to do" list includes:

- "Don't drop the floor under 0.5 for normal play. The
  0.3→0.55 raise was a *playability* fix."
- "Don't raise the plateau past 1.0 globally."

Then 005's proposed table goes to **0.35** at t=1.0 and
**1.05** at t=0.22. Both cross the listed lines. 005
hand-waves with "only briefly" and "use a separate
getWarmth() for the peak." Both handwaves are load-bearing:

- **"Briefly" still flickers at 4× speed.** 010 confirmed
  4× is a live play mode. A window from t=0.92 to t=0.08
  (16% of the day) at 4× is ~3–4 seconds of wall-clock
  darkness per in-game day. That's long enough to bring back
  the "can't see citizens at night" problem the raise was
  preventing.
- **Separate `getWarmth()` can't fix what's named**. 005's
  failure-mode C ("sky and ground are uncoupled — you never
  see warm ground with warm sky") is specifically about
  *ground luminance peaks not aligning with sky peaks*. A
  color-only `getWarmth()` doesn't couple peaks — it just
  tints a flat plateau warmer. Warm-tinted flat is still
  flat. The disease and the cure are in the same variable.

005 can't have it both ways: either the luminance shape
matters (breaking the constraints) or it doesn't (the
separate-function workaround is a no-op).

### 2. The perceptual diagnosis points at the wrong target

005's strongest moment: *"the luminance delta gets
perceptually swallowed under the same blue cast. Dusk and
midday frames look like the same moment because they share
the same dominant color, just at different intensities."*

This correctly identifies the cause. But 005's proposed
fix is the luminance curve, not the tint. If the tint is
what's swallowing the delta, re-shaping the luminance
doesn't help — you could ship 005's full proposal and
midday-vs-dusk would STILL look alike, because the same
blue multiply still dominates both.

The cheaper, more surgical fix: change the tint itself so
its hue shifts with time of day. Current code at
`render.js:2518`:

```
tintR = 255 - cappedDarkness * 160
tintG = 255 - cappedDarkness * 150
tintB = 255 - cappedDarkness * 100   ← B drops least → blue cast
```

The tint is uniformly blue because B is always least-darkened.
If the coefficient profile changed with `dayT` — say, more
red-biased at dusk, more blue at midnight — the midday and
dusk frames would carry different color casts, and the 19%
luminance delta would read against *different* backgrounds
instead of the same one. That's a ~10 line change vs. a
whole-curve re-authoring.

### 3. 010 is evidence against 005's winter diagnosis

005 argued winter reads "underexposed" because real winter
has a lower, dimmer sun; the fix is seasonal coupling.

010 played winter and reported winter *still* reads
moody-dark — even though 004 proved the daylight curve is
byte-identical across seasons. The remaining dark quality
must come from somewhere else. Two candidates:

- `SEASONS.winter.tileShift = [-10, -5, +15]` — pulls red and
  green down, pushes blue up on every tile.
- The multiply-blue overlay (same as failure above) stacking
  with the winter tileShift.

Neither of these is in the lightCurve. Re-shaping luminance
wouldn't touch either. **005's proposed seasonal coupling
treats a symptom that lives in a different file.**

### 4. Sky leads ground is physically correct

005 calls sky-gradient-peaks-without-ground-peaks a
"failure mode." But that's how real sunrise and sunset
actually work: atmospheric scatter lights the sky minutes
before direct sunlight reaches the ground. A game that
warms sky and ground simultaneously reads as theatrical
stage lighting, not a sunrise.

The feature 005 names as a bug is the one physical detail
this game's sky-gradient got right. I'd leave it alone.

### 5. Gameplay vs cover-art is a real dichotomy here

005 treats "legibility" and "beauty" as both-and. At 15 fps
4×-speed play (010 confirmed this IS the dominant mode), no
player is staring at a frame for warmth. They're scanning
citizens, resource tickers, build queues, the raid
countdown. The current flat-plateau curve is *correct for
a city-builder at gameplay pace*.

If we want cover-art quality, the correct surface is a
dedicated photo-mode — which is already in ideas.md (003,
007). A photo-mode can toggle a different curve freely
because gameplay isn't happening during a screenshot.

## so what should happen with the lightCurve?

My recommendation: **leave the luminance curve alone.
Re-spend the attention budget on two cheaper wins.**

1. **Hue-vary the tint overlay** so midday-blue and dusk-
   blue aren't literally the same color. Fixes failure-mode
   A without touching getDaylight. ~10 lines.
2. **Photo mode (ideas.md 003)** — toggle UI hiding + a
   more dramatic luminance curve + sky-gradient override,
   independent from gameplay. This is where the cover-art
   argument pays off without hurting 4×-speed play.

The sky-ground peak alignment argument (failure-mode C) is
the one part of 005's thesis I'd leave open. IF photo-mode
lands, its curve can align with the existing sky peaks;
that's a genuine both-and. But it should be opt-in, not a
gameplay default.

## what this review does not cover

- Whether the current `tileShift` per-season values are
  well-tuned. 010 suggests winter is over-cooled but this
  review didn't measure.
- The night-floor value (0.55) itself. I defended
  *not dropping* it; I did not argue *not raising* it.
  Could be 0.6 is even better. Not measured.
- Whether a future `the-profiler` tick should instrument
  the tint overlay the same way 004 instrumented daylight.
  That would give a second numeric axis for future
  arguments. (Adding idea.)

## ideas generated → ideas.md

- Hue-vary the multiply-tint overlay across dayPhase: same
  darkness magnitude but different color bias at
  midday/dusk/night, so luminance delta reads against
  changing hue. ~10 lines in render.js:2518.
- `G.debug.tintCurve()` sampler — sibling to 004's
  lightCurve — for measuring the tint overlay numerically.
- Photo mode scope-bound curve: a separate `getPhotoDaylight()`
  used only when `G.photoMode === true`, so the flat-plateau
  gameplay curve and a dramatic curve can coexist.

## verdict

005's observations are strong. Its proposed fix is wrong.
Take the observations, keep the curve, fix the tint. This
review is the register for: **if iteration 012 or later
goes to re-shape getDaylight, re-read this first.**
