## existing mobile surface

### touch handlers (input.js:266-309)

- **touchstart**: single-tap places building if one is
  selected; else starts drag (camera pan). Two-finger touch
  records pinch distance.
- **touchmove**: single-touch drags camera; two-finger pinch
  zooms. `e.preventDefault()` to avoid page scroll.
- **touchend**: clears drag state.
- **initAudio** on first touch (standard iOS autoplay policy
  workaround) ✓

### viewport meta (index.html:5)

```html
<meta name="viewport" content="width=device-width,initial-scale=1.0,user-scalable=no">
```

**`user-scalable=no`** — HIGH concern (below).

### media queries (3 total)

```css
@media (max-width:1400px) { .res-lbl{display:none} }
@media (max-width:600px)  { #info-panel {...} }
@media (max-width:600px)  { #hud, .build-btn, #missions,

```

The 600px breakpoint shrinks HUD, build buttons,

## HIGH — user-scalable=no blocks accessibility pinch-zoom

Viewport meta sets `user-scalable=no`. This blocks native
browser pinch-zoom — which many players (vision-impaired,
casual reading) rely on to enlarge text. **Apple's WCAG
guidelines explicitly call this out as an accessibility
violation.** Modern iOS/iPadOS already ignores it on many
sites to preserve zoom regardless.

The author's intent was likely to prevent double-tap zoom
during gameplay (you want touch → place, not zoom). Valid
UX goal but the accessibility cost is real.

**Fix options:**
- (A) Drop `user-scalable=no`. Accepts that double-tap may
  zoom; tolerable because the game tap is single-tap +
  long-press for building placement.
- (B) Replace with `maximum-scale=5, minimum-scale=1` —
  allows zoom but bounded.
- (C) Keep `user-scalable=no` but add an in-game
  "accessibility mode" toggle that removes it.

(B) is the WCAG-friendly middle ground. 1-line change.

## MEDIUM — three panels dominate 390px portrait

At 390px viewport width:
- `#minimap` — 220×220 (post-600px break: 150×150) — fixed
  bottom-right. 150px = 38% of viewport width.
- `#missions` — 220px wide (post-break: 170px) — fixed
  top-right. 170px = 44% of viewport width.
- `#build-bar` — full-width horizontal scroll at bottom.

Combined: minimap (bottom-right 150px) + missions (top-
right 170px) + build-bar (bottom, full width but ≤ 6rem
tall) leaves a center-screen diamond of usable canvas maybe
**200×500 pixels** on a 390×844 screen. Enough to see ONE
tile clearly, not enough to see the settlement.

**Fix options:**
- (A) Hide minimap on <500px viewport (`display:none`) —
  player navigates by drag/pinch.
- (B) Make missions collapsible (click header to fold).
- (C) Make minimap + missions tabs in a single bottom
  sheet (swipe up to expand).

(A) is easy; (B) is small scope; (C) is a real mobile UX
redesign.

## MEDIUM — build-bar x-scroll has no scroll indicator

`#build-bar` uses `overflow-x:auto` with a 4px scrollbar.
On touch devices, scrollbars are usually hidden until
scroll-in-progress. A first-time mobile user may not know
the build bar scrolls — they'll see the first 4-5 buildings
and think that's all.

**Fix:** fade gradient on right edge of build-bar when
content overflows (the 042 pattern from research panel).
Same `_wireScrollCue` helper could work. ~5 lines.

## MEDIUM — no touch-equivalent for tooltip discovery

Desktop: hover-to-see tooltip (titles on HUD buttons, info
on tile decorations). Mobile: no hover. Player taps and
either:
1. Does an action (if tappable)
2. Sees nothing (if the tap did nothing)

14/39 buttons have titles. Mobile players never see any of
them. This duplicates 027's desktop hit-box finding from a
different angle.

**Fix:** long-press (~500ms) shows tooltip overlay. Needs a
touch-timer pattern. ~20 lines.

## LOW — title-screen max-width 480px on 390px viewport

`.title-content` has `max-width:480px`. On a 390px viewport
the content will WIDEN past the viewport OR the layout will
overflow scrollable. Title button padding + buttons-per-row
likely overflows the 5-scenario row.

**Fix:** dedicated `@media (max-width:480px)` block
shrinking `.title-content` + stacking scenario buttons
vertically. ~8 lines.

## LOW — no haptic feedback on placement

`navigator.vibrate(10)` on successful placement would
provide mechanical feedback. Small polish. ~1 line.

## LOW — no swipe gestures

No swipe-to-open-panel (e.g., swipe from right for
chronicle). All panels are icon-tap-to-open. A dedicated
mobile UX would benefit from gesture shortcuts. Out of
scope for a review tick; would be a larger overhaul.

## honest conclusion

**Realm is mobile-usable but not mobile-designed.**
- Touch handlers work for core gameplay (tap place, drag
  pan, pinch zoom).
- Viewport is responsive with 2 breakpoints covering phones.
- Key panels shrink appropriately at <600px.
- BUT: `user-scalable=no` is an accessibility issue;
  tooltip discovery is broken; screen real-estate at
  390px is dominated by panels; touch-friendly
  conveniences (long-press, haptic, swipe) are absent.

This is "works on mobile" territory, not "designed for
mobile." Most browser-first indie city-builders stop at
this level. A mobile-first revamp would be its own
multi-tick thread.
