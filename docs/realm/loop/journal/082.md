2. Override: `G.kingdomName = 'CompletionistTwo'`
3. Unlock all 13 researchable techs, boost all
   resources to 9999

   tiles (spiral search from center)

6. Sample chronicle + flags + named characters

manually), 082 drives `simTick` through the

updateResearch / checkStoryBeats` — everything.

053's programmatic run which needed zero real
time because it never actually ran the full

## HIGH finding — raid-tag chronicle pollution

**197 of 299 chronicle entries** (66%) are
tagged `raid`. But only ~8-10 are actual
memorable raid events. The rest are UI-toast
passthroughs that 076 misclassified as LOW.

Breakdown by prefix (top raid-tagged entries):

```
"⚠️ Scouts report raiders approaching..."     40×   (economy.js:371)
"⚠️ Raiders approach from the darkness..."    40×   (economy.js:377)
"⚔️ RAID: 3 raiders approach!"                36×   (economy.js raid spawn)
"⚠️ No defenders! Build walls, barracks..."   36×   (economy.js danger warning)
"Your defenders move to intercept."             3×
"⚔️ RAID: 2 raiders approach!"                 1×
"The first raid is turned back..."              1×   (story.js firstRaid)
"Raid on day 30 repelled..."                    1×   (resolution)
"Raid on day 38: the village was razed..."      1×   (resolution)
```

Each raid cycle produces ~5 chronicle entries,
4 of which are pre-raid UI warnings. 40 raids
× 5 entries = 200 entries — matches the 197
observed.

**076 audited 19 notify callers and classified
these as "LOW correct" because raid-related
tagging is correct semantically.** 082's data
shows this was wrong. The semantic tag IS
right; the volume is catastrophically high.
Each raid cycle floods the chronicle with
5 near-identical tag:raid entries.

**Filtering chronicle by tag:raid** (via 078's
UI filter) shows 197 entries, 4 of which are
real narrative memory. The filter is useless.

### fix path

Apply `{ chronicle: false }` to the 4 notify
calls that produce the noise:

- economy.js:371 (scouts-report warning)
- economy.js:377 (raiders-approach warning)
- economy.js raid-spawn `notify('RAID: N raiders')`
- economy.js no-defenders warning

Keep chronicling only the RESOLUTION beat
(story.js:233 firstRaidSurvived + economy.js:428
per-raid report line which already chronicles
via direct call not notify).

Post-fix, each raid would produce ~1-2 chronicle
entries instead of 5. 200-day realm raid count
drops from 197 to ~15-20 real raid memories.

## HIGH finding — once-per-realm beats didn't fire

Nightmare for CompletionistTwo: seeded day 223
(past 200-day run; expected miss).
Stone for CompletionistTwo: seeded day **131**
— SHOULD have fired but didn't.
Dreams: every 10 days from day 10. **Zero
dreams in 200 days.**

Hypothesis: the dawn-gate (`dayPhase > 120
return`) combined with `crossed(60)` sampling in
`simTick` may be skipping the dawn window entirely
under fastForward. When G.speed=60, `simTick`
advances 60 game-ticks per iteration. The
`crossed(60)` trigger that drives
`checkStoryBeats` fires when gameTick crosses a
multiple of 60. With dayLength=3600 and dayPhase
cycling, the dawn window (phase 0-120) lasts
2 `crossed(60)` samples per day — but those
samples may land at phase 60 and 120, NOT inside
the dawn window.

Need to verify. If fastForward reliably MISSES
dawn-gated beats, it's a serious limitation of
the helper.

Filed as a 082 HIGH.

## MEDIUM — year-milestone text has grammar + timing bugs

Small fix; filed.
