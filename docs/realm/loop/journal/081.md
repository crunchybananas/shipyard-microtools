## the fix

`docs/realm/js/main.js` — new `fastForward(days)`
function + wired on `G.debug`:

```js
function fastForward(days) {
  if (!G || typeof G.dayLength !== 'number') return { error: 'game not initialized' };
  if (!Number.isFinite(days) || days <= 0) return { error: 'days must be positive number' };
  const startDay = G.day;
  const targetDay = startDay + Math.floor(days);
  const origSpeed = G.speed;
  G.speed = 60;  // each simTick advances 60 game-ticks
  const maxIters = Math.ceil((days * G.dayLength) / G.speed) + 100;
  let iters = 0;
  while (G.day < targetDay && iters < maxIters) {
    simTick();
    iters++;
  }
  G.speed = origSpeed;
  return { advancedDays, gameTick, iters, day, season, happiness, population, chronicleLen };
}
```

Wired: `G.debug.fastForward = fastForward;` next
to renderBuildingIsolated / dreamLens.

## design choices

- **Temporarily boosts G.speed to 60.** Each
  `simTick` then advances 60 game-ticks instead
  of the default 1-4. This is the same batching
  the existing `gameLoop()` uses when the tab is
  hidden (main.js:440 does `for (let i = 0; i < 60;
  i++) simTick();`). Reusing the pattern.
- **Calls `simTick()` directly** — not replicating
  its tick-update logic. If a future tick adds a
  new update hook, `simTick` picks it up; so does
  `fastForward`.
- **Safety cap `maxIters`** — prevents an infinite
  loop if `G.day` somehow stops advancing.
  Generous (days×dayLength/60 + 100) so legitimate
  long runs don't hit the cap.
- **Input validation**: negative/zero/non-number
  returns `{ error: ... }` without side effects.
  Lets callers defensively check.
- **Return value is a state snapshot**: day,
  season, happiness, population, chronicle length.
  Useful for follow-up assertions in audit ticks.

Let me recalc. 069's test was at G.speed=4 and
chrome-mcp's setTimeout-throttled `for (...) simTick()`
loop. Real-time visible. 081 is a synchronous

The BIG win isn't raw throughput. It's **zero
real-time cost**. A test tick can call
`fastForward(200)` and get state back
synchronously, no waiting for 3860ms of real
time. The 3860ms IS the real time; the script
can simply `await` nothing because `fastForward`
is synchronous.

In practice the elapsed time only matters for
tests scripted in a single `javascript_exec`
call — where we WERE waiting for setTimeout-
throttled ticks. Now the same 200-day result
returns in ~4 seconds of blocking JS instead of
~100 seconds of real-time-wall-clock-setTimeout.

Still a meaningful improvement. Ratio closer
to ~25× for chrome-mcp workflows.

## what I did NOT do

- **Skip particle/smoke updates** for speed.
  They're cheap per tick and preserve full
  simulation fidelity. A `{ noParticles: true }`
  option could be added later.
- **Suppress rendering inside** the helper. There
  IS no rendering during fastForward — `simTick`
  doesn't render, only updates state. If a caller
  wants a frame after, they can call
  `window.forceRender()` themselves.
- **Log progress**. Just returns final state.
  Callers can compute delta from before/after
  snapshots.
- **Accept `{ startDay: N }` option** to start
  from a specific day. Callers set `G.day`
  directly if they need that.
- **Restore G.chronicle between calls.** If the
  helper gets called 3 times in a test, entries
  accumulate. Callers can `G.chronicle = []`
  between calls.
