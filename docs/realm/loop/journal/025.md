## the bug (024's repro, for the record)

1. Place a new-type building (e.g. bakery).
2. Story beat fires next tick — `firstBakery` flag set,
   chronicle gains *"First loaves cool..."* entry.
3. `undoLastBuild()` removes the building.
4. Player rebuilds a bakery — no new chronicle entry.

Net: chronicle narrates an undone event; actual first-
build gets no narrative acknowledgment.

## the fix

Two small edits in `js/economy.js`.

**At push (line 72–74)** — change the undo-stack entry
from a raw `b` to a wrapper:

```js
G._undoStack.push({
  b,
  flagsSnapshot: { ...(G.storyFlags || {}) },
  chronicleLen: (G.chronicle || []).length,
});
```

Shallow clone of `storyFlags` is sufficient — it's a
flat object keyed by flag name. Chronicle length is
captured because restoring flags alone would leave a
"first loaves cool" entry in the chronicle even after
undo.

**At pop (`undoLastBuild`)** — unwrap defensively, then
restore:

```js
const entry = G._undoStack.pop();
const b = entry && entry.b ? entry.b : entry;
// ... existing undo logic ...
if (entry && entry.flagsSnapshot) G.storyFlags = entry.flagsSnapshot;
if (entry && typeof entry.chronicleLen === 'number' && G.chronicle) {
  G.chronicle.length = entry.chronicleLen;
}
```

The defensive `entry && entry.b ? entry.b : entry`
unwrap handles any legacy raw-`b` entries (none expected
— undo stack is in-memory only, not serialized to save
per `save.js` grep), but costs nothing to keep.

`node --check js/economy.js` passes.

## verification (chrome-mcp, 024's exact repro)

```
after place   → buildings: 1, chronicle: 1, firstBakery: true
after undo    → buildings: 0, chronicle: 0, firstBakery: false  ← WAS: 0,1,true (the bug)
after rebuild → buildings: 1, chronicle: 1, firstBakery: true
                lastChronicle: "The first loaves cool on the baker's rack..."
```

All three invariants hold:
1. Undo fully reverts the place — flag, chronicle, and
   building all rewound.
2. Rebuild re-narrates correctly.
3. Normal placement (no undo) path unchanged — the new
   wrapper shape is invisible to callers.

## edge cases considered

**Multiple placements in the undo window.** If a player
places bakery #1 (first beat fires), then places bakery
#2 (no beat — flag is set), then undos:

- Undo of #2 restores flags to `{firstBakery:true}` and
  chronicle to len=1. Correct — #2 wasn't a first.
- Second undo would restore flags to `{}` and chronicle
  to len=0. Correct.

Each push captures the *pre-place* flag state, so each
undo walks the history back step-by-step.

**Chronicle entries from other systems (raids, seasons)
between place and undo.** If a season changes or a raid
fires between placing a building and undoing it, the
chronicle-length truncation would lose those entries too.

This is the one honest tradeoff. Undo is user-driven
(Ctrl+Z), typically <1s after placement, so inter-event
windows are rare. Considered tracking entry authorship
(which-system-added-which) but that's a much bigger
refactor. Accepting the tradeoff per 024's option (a)
scope.

**Buildings that shouldn't re-fire beats on undo → redo.**
e.g. castle. Placing a castle sets `G.won = true` and
schedules the victory screen; an undo would... not
actually un-win? Checking line 82-85 of economy.js: yes,
`G.won = true` stays set on undo. That's a separate,
pre-existing bug not in scope. Filing as idea.

## what I did NOT do

- Refactor the undo system to a full command-pattern.
  The wrapper approach is minimally invasive.
- Fix the `G.won` leak on castle-undo (noted as idea).
- Extend undo-aware snapshots to other game state
  (resources, pop, defense). These already roll back
  in `undoLastBuild` via the manual unwinding. The
  snapshot-approach could arguably replace that, but
  that's a larger refactor.
