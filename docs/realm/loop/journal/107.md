## HIGH — H-key collision silently breaks photo-mode hotkey

`input.js:319`:
```js
if (e.key === 'Home' || e.key === 'h') {
  ...
  G.camera.zoom = 1.3;
  return;
}
```

`input.js:388`:
```js
if (e.key === 'h' && !e.ctrlKey && !e.metaKey) {
  if (window.togglePhotoMode) window.togglePhotoMode();
  return;
}
```

Both handlers are in the SAME `keydown` listener. The first
`return` exits. So pressing `h`:

1. Line 319 matches → recenter camera → `return`
2. Line 388 never executes

**Photo-mode toggle via H KEY HAS NEVER WORKED** since 035
shipped it (~72 ticks). Photo mode can only be entered/
exited via:
- `window.togglePhotoMode()` in the console
- Whatever button invokes it (no on-screen button found)

And critically: the photo-mode badge displayed at bottom of
screen says **"Photo Mode — press H to exit"** — **a broken
instruction that has misled every player who enters photo
mode for 72 ticks**.

**Impact:** users enter photo mode (somehow), read the
"press H to exit" hint, press H, camera recenters instead,
and photo mode STAYS ON. Only escape is console or close
tab. 035/098/099 all relied on console toggling — the user-
facing UX has been broken without anyone noticing because
the loop does all photo-mode work via console.

**Fix:** drop `|| e.key === 'h'` from input.js:319. Leaves
`Home` as the recenter key (standard expectation anyway).
One-token delete. Photo-mode H hotkey starts working.

```js
// line 319, was:
if (e.key === 'Home' || e.key === 'h') {
// fix:
if (e.key === 'Home') {
```

## MEDIUM — no keyboard building PLACEMENT path

Selecting a building type with `1-9` or `B` works. But
placing it requires a click on the canvas. A keyboard-only
player can:
- Open research panel ✓
- Pan camera ✓
- Zoom ✓
- Select building type ✓
- **Place a building: impossible**

To finish even the 30-second intro, a player must mouse.

**Fix options:**
- (A) Enter places at camera-center tile.
- (B) Arrow keys move a "placement cursor" diamond on the
  grid; Enter places there; Escape cancels.
- (C) Document as a limitation (mouse-first by design).

(C) is honest but documentation-only. (B) is the real
keyboard-player fix — ~30-50 lines, medium scope.

## MEDIUM — poor ARIA coverage on menu buttons

DOM audit showed:
- 39 total buttons
- 14/39 have `title` (mouse-hover tooltip)
- **4/39 have `aria-label`**
- 0 have negative tabIndex (good — all in tab order)

Screen readers read emoji-only buttons (`🔊`, `🎵`, `🏆`)
as "speaker", "musical note", "trophy" — reasonable but not
precise. Adding `aria-label="Toggle ambient sound"` etc.
improves screen-reader UX considerably.

**Fix:** add aria-label to all HUD icon buttons. ~10 buttons,
~10 lines. LOW-MEDIUM.

## MEDIUM — B-cycle has no visual indicator

Pressing `B` cycles `G.selectedBuild` through building types.
`renderBuildBar()` is called but the build bar doesn't
visibly highlight the currently-selected type from keyboard
(clicking DOES highlight via CSS hover/active states).

A keyboard player cycling with B has no feedback about
which type is selected without watching the placement
preview (which requires mouse position).

**Fix:** render a visible "current selection" border on the
build bar card matching `G.selectedBuild`. CSS-only if the
card's DOM has a class marker. ~3-5 lines.

## LOW — inconsistent shortcut documentation

Button titles sometimes include shortcut in parens:
- `"Research — unlock new buildings and upgrades (R)"` ✓
- `"Music on/off (M)"` ✓ — but M doesn't toggle music
- `"Happiness — view settler mood breakdown"` (no shortcut)
- `"Ambient sound on/off"` (no shortcut)
- `"Achievements"` (no shortcut)

Players discover shortcuts by hovering (mouse). Keyboard
players never see these titles.

**Fix:** a "keyboard shortcuts" section in the help overlay
(`?` key) — index.html already has `#help-overlay`.

## LOW — M key dual-use conflict

Button title on music button says `"Music on/off (M)"`, but
`M` actually toggles minimap. Music has no keyboard
shortcut. Title is misleading.

**Fix:** either remove `(M)` from the title or actually wire
M to toggle music.

## LOW — L and Tab undocumented

`L` toggles log; `Tab` cycles panels. Both undocumented
anywhere outside help-overlay content.

## the audit's honest conclusion

**Realm is mostly keyboard-accessible** — you can do
everything but place a building. That's a significant
limitation (the game IS placement), but most UI operations
(panels, research, speed, camera, zoom, chronicle) all
work.

The HIGH finding (H-collision) is a real 72-tick-old bug
that a 2-token fix closes. The MEDIUM findings (ARIA,
B-feedback, placement) are known-scope fixes. The LOWs are
polish.
