## the build

### Save side

```js
state.realmEnded = !!G.realmEnded;
```

Placed alongside `state.savedAt` line 85 (139's
addition pattern). Persists post-end mode across
reload.

### Load side

```js
G.realmEnded = !!s.realmEnded || !!(s.storyFlags && s.storyFlags.realm_fell);
```

Placed alongside `G.storyFlags` line 175. **Two
fallback paths**:
1. `s.realmEnded` — the explicit save field
   (post-197 saves)
2. `s.storyFlags.realm_fell` — derive from
   the existing narrative flag (legacy
   pre-197 saves OR any path where the
   beat fired but the explicit field
   wasn't set yet)

Either path produces correct post-end-mode
state on load.

10 LoC additive (5 LoC code + 5 LoC comments).

## why this shape

- **Two-path load is defensive.** A future
  consumer of `G.realmEnded` (render desat,
  audio fade, etc) will see correct state
  whether the save predates 197 or postdates.
  Avoids a "load fall-back narrative replay"
  scenario.
- **`!!` boolean coercion** prevents
  type-mismatch bugs. Save layer normalizes;
  load layer normalizes; consumers see clean
  boolean.
- **Save side is minimal.** Single `state.X = !!G.X`
  line. Matches the 139 savedAt pattern (single
  line in same block).
- **Load side derives from existing flag if
  needed.** Don't require the field; recover
  it. Backward-compatible by construction.
- **Save field NAME matches the runtime
  name.** `state.realmEnded` ↔ `G.realmEnded`.
  No semantic translation.
- **Closes the 192 filing exactly.** 192's
  filed text: "save.js persistence:
  state.realmEnded = G.realmEnded; on save;
  G.realmEnded = !!s.realmEnded; on load.
  ~2 lines." Shipped as 2 effective code
  lines (the comment lines are explanatory
  not functional). Plus the legacy-derive
  fallback as defensive programming.
