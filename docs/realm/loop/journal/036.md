# 036 — the-fixer (sticky raid banner + end-toast)

**Challenge:** the-fixer — accepted. Closes 010's
biggest-unfixed finding: *"raids happen off-screen
without ceremony."* The mechanic exists (enemies
spawn, fight, die) but the player never sees it
happen at a glance. This tick adds the UI that turns
ambient combat into a visible emergency.

Second fix under the new cadence (034), continuing
the "build for progress" direction 035 started with
photo mode.

## what shipped

**(1) Sticky in-progress banner** — a `#raid-banner`
DOM element at top-center-below-HUD, shown while
`G.enemies.length > 0`. Red-tinted, semi-transparent,
non-modal (pointer-events: none). Text: *"⚔ Raid in
progress — N raiders"* updating each tick.

The banner is driven by a new `updateRaidBanner()`
function registered via `registerUpdater()` alongside
the existing `updateRaidChronicle`. When
`G.enemies.length === 0`, the element hides via
`display: none`.

**(2) End-of-raid summary toast** — extended the
existing `updateRaidChronicle()` in enhancements.js
to ALSO fire a `notify(msg, 'event')` (or `'danger'`
if village razed) alongside the chronicle entry.
Reuses the same composed message ("Raid on day N
repelled — X foes slain, Y of ours lost.") that the
chronicle already writes. `chronicle: false` flag on
notify suppresses re-chronicling.

**(3) Photo-mode hide** — added `#raid-banner` to
035's `body.photo-mode` CSS rule so it's hidden
during screenshots, consistent with the rest of HUD
chrome.

## verification (chrome-mcp, end-to-end)

Forced a raid with the standard test pattern (jump
day to nextRaidDay, add Barracks + military techs so
014's raidsSurvived gate passes, fire `checkRaids`):

```
enemies spawned: 2
banner DURING: display=block, text="⚔ Raid in progress — 2 raiders"
[kill all enemies]
banner AFTER:  display=none
toast:    "Raid on day 37 repelled — 2 foes slain, none of ours lost."
chronicle: ["⚔️ RAID: 2 raiders approach!",
            "⚠️ No defenders! Build walls, barracks, or towers.",
            "Raid on day 37 repelled — 2 foes slain, none of ours lost."]
```

All three channels (sticky banner → summary toast →
chronicle entry) work together. Screenshot confirms
banner position is below the HUD and above the
settlement, not overlapping the mission panel or
tutorial tip.

## what I noticed mid-tick

**Banner requires `G.speed > 0`.** `simTick()` in
main.js returns early when paused (`if (G.speed <= 0)
return;`), so updaters don't fire. First verification
attempt showed the banner stuck at `display:none`
even with enemies present — because I'd paused before
firing the raid. This matches real-player behavior:
if you pause during a raid, the banner's text won't
update mid-pause. Acceptable for now; a future tick
could move the banner update into a separate
always-running loop if this becomes a complaint.

**`updateRaidChronicle` only fires on SURVIVED raids.**
Post-014, `raidsSurvived` is gated on `hasDefense`.
For a defenseless settlement where raiders kill some
citizens and leave, neither the chronicle entry nor
my new toast fires. Arguably correct (no "victory"
to announce), but it also means the worst-case raid
outcome has the LEAST UI feedback. Worth a future
refinement — a separate "raiders withdrew" message
for no-defense scenarios. Filed as idea.

**Banner styling took a moment to settle.** First try
used a neutral-dark background that blended into the
sky gradient at dusk. Switched to `rgba(135,20,20,0.88)`
— deep red, more than adequate contrast.

## what I did NOT do

- **Auto-pause on first raid** (010's original cluster
  included this). Adding a forced pause breaks speed-
  running and idle play; ideally it's a user-toggled
  preference. Left for a future tick.
- **Persistent on-map destruction markers** (030
  idea). Different concern — bodies/ruins on tiles.
  Out of this tick's scope.
- **Split chronicle `event` vs `danger` tag.** Used
  `event` by default; `danger` only if village razed.
  Matches the tone-of-voice in the existing chronicle
  text.
