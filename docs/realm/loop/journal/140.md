## why silent-module now, not the full migration

- **130's audit recommended the variables.** Defining
  them is the easy half. Migrating ~100 font-size
  declarations is the time-consuming half.
- **Silent-module maxim: "plumbing that pays rent
  later."** A future tick touching (say) `.build-btn`
  can change `font-size: 0.7rem` → `font-size:
  var(--fs-sm)` in passing. No dedicated migration
  tick required. Incremental drift-reduction rather
  than one-shot burst.
- **Zero behavior change.** The variables are defined
  but nothing uses them yet. Existing CSS unchanged;
  every `font-size: 0.7rem` still resolves to
  `0.7rem`. Safe to ship without verification.
- **Documented token choices.** The CSS comment block
  records the mapping from observed values to tokens.
  Future ticks can cite the comment when deciding
  which token fits.

## what remains open

- **130 HIGH phase 2** — migrate existing ~100
  font-size declarations to use the tokens. Can be
  done all-at-once (refactor-only tick, ~1 hour of
  CSS editing) OR piecemeal (adjacent to future
  typography-touching ticks).

I recommend piecemeal. One-shot migration risks:
- breakage at the "close enough" boundary (is
  0.78rem → `--fs-base` (0.82) or `--fs-sm` (0.75)?)
- test coverage needed to catch visual regressions

Piecemeal avoids both — each migration happens in
context with a tick that's already verifying the
affected area.
