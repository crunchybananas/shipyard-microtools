## the build

`docs/realm/js/render.js` — three changes (~70 LoC).

### 1. `_VARIANT_PALETTES` declaration table

Maps sprite type → token group → array of
parallel color tuples. Index 0 is the SOURCE
(must match the SVG file exactly); indices 1+
are variant TARGETS. Kingdom-hash mod N picks
which target replaces the source.

Initial entry: church → glass with 4 palettes:
- 0 blue (default; `#a8d0ff/#4488cc/#1a3866`)
- 1 red (clay-fire; `#ffb0a8/#cc4444/#661a1a`)
- 2 amber (sunset; `#fff0a8/#cc8844/#664422`)
- 3 green (moss; `#a8ffd0/#44cc88/#1a6638`)

### 2. `_kHash` + `_applyVariants`

Tiny `Math.imul`-based string hash mirroring
`_dreamHash` from story.js without the import
(avoids circular-dep risk through the render →
state → story chain). `_applyVariants` walks
the palette table and string-replaces source
colors with target colors when the kingdom
hash picks a non-zero variant.

### 3. Async fetch path in `_loadSprite`

Loader signature changed: `_loadSprite(type,
kname)`. Cache key now `${type}__${kname || ''}`.

If the type has variants AND kname is set:
- `fetch('assets/sprites/${type}.svg')`
- `r.text()` → applyVariants → Blob → URL
- `new Image()` → onload sets cache

Else: fast-path direct `img.src` (preserves the
fast load path for non-variant sprites).

`drawSpriteIfReady` passes `G.kingdomName` to
the loader. `window.__realm.toggleSVG` preloads
all 11 sprites with the current kingdom name.

`node --check js/render.js` → OK.

## what I did NOT do

- **Per-INSTANCE (per-building) variation.**
  Each church in Avalon looks the same as
  every other church in Avalon. Per-building
  uniqueness would need building-id hashing.
  Filed; lean against — too noisy.
- **CSS custom properties.** The img-tag SVG
  path can't expose CSS to the parent doc.
  Switching to inline SVG would enable CSS
  vars but adds DOM-layer complexity not worth
  the gain. Static string replacement is the
  right shape.
- **Variant entries for other sprites.** The
  framework supports them; adding entries is
  ~5-10 LoC each (define palettes; ensure SVG
  source colors match). Filed:
  - tavern (175): sign palette + emblem swap
  - blacksmith (176): forge fire palette
    `--fire-core`/`--fire-mid`/`--fire-outer`
  - market (179): 4 awning palettes
  - house (173): tile color (red/slate/moss)
- **Cache invalidation on kingdomName change.**
  Realms can't normally change their kingdom
  name mid-game; cache leak is bounded by
  number-of-realms-played-this-session. If
  it becomes a concern, add a clear() on new-
  game.
- **Hash distribution test.** With 4 palettes,
  4 realms producing 3 distinct (1 collision)
  is a small sample. A 100-realm test would
  validate distribution; deferred.
