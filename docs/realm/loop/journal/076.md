# 076 — the-archivist (duplicate-toast audit)

**Challenge:** `the-archivist` — accepted. 070 filed
this as a follow-up: "Audit other toast+chronicle
double-fires besides season." 070 fixed the season
case; 076 audits the rest.

Closed in 075's narrative-surfaces doc:
`notifications.js:68` routes notify() calls with
type ∈ {event, danger, mission} to chronicle()
automatically (unless `{chronicle: false}` is
passed). This makes any callsite that ALSO has a
direct chronicle() call a potential duplicate.

## method

1. `grep -rn "notify(" js/ | grep -v notifications.js`
   → 19 callsites
2. For each, check whether a direct `chronicle()`
   call appears within ~5 lines (same conceptual
   event)
3. Check the notify type — determines whether the
   toast will also write chronicle
4. Classify HIGH (duplicate), MEDIUM (wrong tag),
   LOW (probably-intended)

## HIGH findings — confirmed duplicates

### 1. combat.js:133-134 — raider-kill death

```js
try { notify(`${c.name || 'A settler'} was slain by raiders!`, 'danger'); } catch(_e){}
try { chronicle(`${c.name || 'A settler'} fell to raiders. Their name joins the stone.`, 'death'); } catch(_e){}
```

Each raider-kill produces **two** chronicle entries:
- `raid` tag via notify route (danger → raid in
  tagMap at notifications.js:67)
- `death` tag via direct chronicle

Same conceptual event. Duplicate.

### 2. economy.js:337-338 — starvation death

```js
notify(`${c.name} has died of starvation!`, 'danger');
try { chronicle(`${c.name} perished from hunger. The realm mourns.`, 'death'); } catch(_e){}
```

Same pattern. `danger → raid` via notify, `death`
direct. Two entries per starvation.

### 3. economy.js:162 + story.js pop-thresholds

```
economy.js:162  notify(`🎉 Population reached ${m.label}!`, 'event')
story.js:222    chronicle('Ten souls now call this land home.', 'milestone')  (at pop=10)
```

Population crossings at **10/25/50/75** produce two
chronicle entries: `event` from the notify route,
`milestone` from story.js's popCheck table. story.js
has pop=100 too; economy.js stops at 75, so pop=100
is single-entry.

Four duplicate callsites per realm (one per
threshold).

## MEDIUM findings — wrong-tag chronicle writes

All type='danger' notify calls that AREN'T about
raid combat. These write chronicle with tag='raid'
due to the tagMap at notifications.js:67 — wrong
label for the event:

### 4. ui.js:13 — food-low warning

```js

```

Writes chronicle tag='raid'. A food shortage isn't

with non-raid content.

### 5. economy.js:344 — settler leaves

```js

```

Tag='raid'. Really a `death` or `event` would fit
better.

### 6. economy.js:577 — upgrade cost-fail

```js
notify(`Not enough resources to upgrade! Need: ${costStr}`, 'danger');
```

Tag='raid'. Pure UI failure message; shouldn't be
in chronicle at all.

### 7. economy.js:693 — building burned

```js
notify(`🔥 ${BUILDINGS[b.type]?.name || b.type} burned down!`, 'danger');
```

Tag='raid' for a fire. Could be more specific
(event or a new tag).

### 8. main.js:251 — trade insufficient

```js

```

Tag='raid' for a trade-UI failure. Shouldn't
chronicle at all.

## the pattern

070 fixed the season case by passing `{chronicle:
false}` to the notify. The same fix works for
nearly all HIGH/MEDIUM cases here:

- Cases **1-3** (HIGH, real duplicates) need the
  opt-out so the two callsites don't fight.
- Cases **4-8** (MEDIUM, wrong-tag) need the
  opt-out because a UI warning really shouldn't
  be in the chronicle at all. "Food running low"
  isn't memory — it's a live prompt.

Alternative approach for 4-8: widen the tagMap at
`notifications.js:67` so type='danger' can produce
tag!='raid'. But that requires the caller to know
the tag; better to just let them opt out.

## why 075's narrative-surfaces doc wins

075's "notify+chronicle at same callsite → pass
`{chronicle:false}`" invariant was written FROM
070's pattern. This audit confirms that invariant
applies in at least 10 places. The doc was
predictive.
