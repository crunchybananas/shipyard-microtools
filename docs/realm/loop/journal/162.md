## what SVG enables here that canvas struggles with

- **`pattern` for stone-block texture.** Tileable
  via `patternUnits=userSpaceOnUse`. The canvas
  version `drawCastle` writes 2 nested `for` loops
  with conditional staggered offsets and 30+ stroke
  calls; SVG expresses the same texture as a single
  `<rect fill="url(#stoneBlocks)" opacity="0.85"/>`.
- **Gradients for stone-and-light gradients on
  every face.** Tower body, keep body, gold spire,
  red roof — each gets its own `linearGradient` /
  `radialGradient`. Canvas needs ~5-line
  `createLinearGradient` chains per face;
  reusability is poor.
- **Window halo glow as `radialGradient`** —
  declared once in `<defs>`, instanced 12 times via
  `circle fill="url(#windowGlow)"`. Canvas writes
  `createRadialGradient` 12 times.
- **Crisp at photo-mode zoom (5×).** Canvas pixel-
  stairs at 5×; SVG stays vector-clean. Most
  beneficial for the gold spire finial and the
  cross emblem on the flag — both tiny details that
  blur in canvas at default zoom and still look
  sharp at SVG's max zoom.
- **Authoring cost of decorative detail.** The
  castle has ~100 distinct shapes (mortar lines,
  arrow slits, portcullis bars, merlons, windows,
  keystone, etc). Canvas drawCastle is ~245 lines
  of imperative drawing instructions; the SVG is
  ~190 lines of declarative markup. The gap widens
  as the building gets more complex.

## the canvas-version reference

`drawCastle` lives at `js/render.js:4665-4908` (~245
lines). The SVG matches its FORM (same flank-keep-
roof composition, same pennant colors, same glowing
windows, same gold flag with cross emblem) but
elevates the EXPRESSION (cleaner gradients, crisper
detail, vector-scaling).

Pennant sway is **not** in the SVG — that requires
animation (`<animate>` element or CSS keyframes) and
adds complexity. Static pennants are fine for a
sprite reference; if integration ships, animated
SVG is filed for a follow-up tick.

## what I did NOT do

- **Replace `drawCastle` in the live game.** Same
  sandbox-first principle as 161. Integration is a
  separate tick after 3+ sprites exist (so the
  pipeline pays for itself).
- **Animate the pennants/flag.** Canvas version has
  `Math.sin(G.gameTick * 0.06) * 1.5` flag sway and
  per-tower pennant sway. Static SVG for now;
  animation is an integration concern.
- **Add winter snow cap variant.** Canvas castle
  has a winter cap via `_WINTER_CAPS._default`
  (since castle is height-distinct, the default
  cap geometry works). SVG sprite is summer-only;
  winter overlay is a render-pipeline concern, not
  a per-sprite asset.
- **Day/night tint.** Canvas applies a multiply-
  blend tint over the whole scene at dusk/night.
  SVG sprite stays summer-noon; integration tick
  decides how to composite.
- **Touch the 3D engine.** Still parallel axis;
  separate tick.
