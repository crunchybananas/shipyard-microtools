## method

Loaded realm at http://localhost:8889/, scripted a
canvas pixel sampler that:

- Read `canvas.getContext('2d').getImageData` across
  a 40×20 grid per frame
- Regioned samples into topSky / midGround / bottom-
  Ground / edgeLeft
- Forced `G.dayPhase`/`G.season` programmatically then
  called `window.forceRender()` before sampling
- Frames: 5 time-of-day slices at spring, then 4
  seasons at midday

## the finding (HIGH)

**The sky palette is entirely season-blind.** Midday,
all four seasons:

```
spring  midday topSky: [78, 98, 125]
summer  midday topSky: [77, 97, 124]
autumn  midday topSky: [77, 97, 124]
winter  midday topSky: [77, 97, 124]
```

Distance spring→winter = √3 ≈ 1.7 per 255 → **0.7%
visual distance**. Indistinguishable.

Ground tiles, same four frames:

```
spring  midday ground: [73, 114,  79]
summer  midday ground: [76, 112,  78]
autumn  midday ground: [79, 108,  76]
winter  midday ground: [84, 116,  96]
```

Ground DOES vary — autumn is warmer (R=79, B=76),
winter is blue-tinted (B=96) — but the spread is
also small. Spring/summer ground distance = √(9+4+1)
= 3.7. Visible in side-by-side only.

### why

The sky gradient is defined at `docs/realm/js/
render.js:118-121` as four hardcoded hex constants:

```js
const nightTop = 0x06080f, nightBot = 0x121628;
const dawnTop  = 0x2e2a50, dawnBot  = 0xc6703a;
const dayTop   = 0x8fb4de, dayBot   = 0xd0dbe8;
const duskTop  = 0x3a2a58, duskBot  = 0xd06044;
```

There is no `SEASONS.<season>.skyShift` at all. The
only seasonal modulation in render.js is
`seasonShift = getSeasonData().tileShift` at
`render.js:194`, which is applied via `shiftColor`
at `render.js:218,223` onto tile colors only.

`SEASONS` (`docs/realm/js/state.js:166-179`) defines
four `tileShift` triples:

```
spring tileShift: [ 0, 12,  0]   state.js:167
summer tileShift: [ 8,  5, -5]   state.js:168
autumn tileShift: [15, -5,-10]   state.js:169
winter tileShift: [-5, -3,  8]   state.js:178
```

Loop 017 halved winter from [−10,−5,+15] to
[−5,−3,+8] with good reason (013's "drained" finding).
That fix is still correct at winter's canonical time
(dusk with 012's cool-indigo night-tint). The
un-intended side-effect is that **at midday when the
night-tint is OFF (daylight ≥ 0.95 — render.js:2516
guard), the only seasonal channel is `tileShift`, and
the sky has zero seasonal content** — so winter-midday
reads as "spring-midday with slightly cooler grass."

013 flagged winter-midday as the weakest frame even
after 012+017. This is the underlying reason: at
midday, only ~10% of the rendered area can carry the
seasonal signal (ground tiles), because sky, sky-tint,
and overlay tint are all identical across seasons. And
the ground seasonal shift is already small.

### fix shape (for a future `the-fixer` tick)

One-line addition in SEASONS (state.js:166-179) +
three-line lookup in render.js:118-121:

```js
// state.js: add skyTop/skyBot offsets per season
spring: { ..., skyShift: [  0,  +5,  +5] },  // barely detectable fresh
summer: { ..., skyShift: [ +8,  +5,  -5] },  // warmer yellow cast
autumn: { ..., skyShift: [+15,  -5, -10] },  // amber haze
winter: { ..., skyShift: [ -8,  -5, +15] },  // slight cool blue

// render.js:120 around dayTop/dayBot
const sShift = getSeasonData().skyShift || [0,0,0];
const dayTop = shiftHex(0x8fb4de, sShift);
const dayBot = shiftHex(0xd0dbe8, sShift);
```

Don't apply skyShift to dawnTop/duskTop — those
already carry strong warm/cool signals of their own
and need to override season at horizon hours. Only
apply during the `day` blend coefficient (so season
reads at midday, fades at dawn/dusk).

Magnitude calibration: keep total per-channel shift
≤ 15 so the midday sky still reads as "blue sky" in
all four seasons. The goal is a *tint*, not a re-skin.

## the finding (MEDIUM)

**Dusk's warm signal is bottom-half-only and fragile.**
Sampling the top 5% of canvas at dusk peak (t=0.78):

```
dusk peak upper sky: [38, 29, 51]    (cool purple)
```

But the code's `duskBot = 0xd06044` = [208, 96, 68]
is warm orange — applied only to the gradient's
bottom stop (render.js:139). With the current
isometric camera, the horizon band is typically
clipped in the bottom 15-25% of canvas. A player with
the camera panned to look slightly up will see
virtually no dusk warmth at all — just a cool upper
sky that doesn't meaningfully differ from late
afternoon.

dawn has the same structure:
- `dawnTop = 0x2e2a50` (cool violet) + `dawnBot =
  0xc6703a` (warm orange)
- Same horizon-band fragility

### fix shape

Either:
- widen the warm-bottom band up the canvas (change
  grad stops from 0→1 to something like 0→0.6 for
  dawn/dusk so warm extends higher), or
- add a second "sunset glow" gradient band (there
  IS already a `horizonAlpha` at render.js:2552 — it
  overlays the sky, value ~0.8 × sin, color `255,140,60`
  at dawn — could be tuned up)

Marking for follow-up rather than specifying precisely:
the right tune would want a screenshot-critic tick
looking at 6-8 camera angles.

## the finding (LOW — good signal)

**012's hue-varying night tint is holding.** Dusk
upper sky [38, 29, 51] has R > G (warmer than
neutral), night upper sky [13, 14, 22] has near-
neutral R ≈ G with B highest (cool indigo). The
perceptual distance between dusk and night *when
they're each rendered at their peak* is clear:

```
dusk  t=0.78:  [38, 29, 51]    warm, pink-violet
post  t=0.85:  [24, 20, 34]    cooling
early t=0.92:  [13, 14, 22]    deep cool
night t=0.05:  [13, 13, 21]    deeper cool
```

The 012 hue-lerp math (render.js:2530-2541) does what
it claims: kR/kG/kB interpolate amber→indigo over
t=0.70→1.00. This lighting thread's work from ticks
003-018 is still standing.

## the finding (LOW — consistent darks)

The recurring [0,0,16] pixel bucket dominating dusk
(290/800 samples) and night (286/800) is the water /
off-island dark-indigo fill. It stays within ±2 per
channel across all dim frames I sampled → no
flickering or time-drift. Good — this is the UI's
"dead space" and it's behaving.

## secondary — palette pairs that *don't* clash

Spot-checked what color-eye is supposed to find: pairs
that BLUR. Didn't find anything egregious:

- grass green [64,176,80] vs placement-pulse
  green `#4ade80` = [74,222,128] (render.js:878,
  from loop 009) — distance √(100+2116+2304) = 66.5,
  readable
- roof red `#c83232` (render.js:618) vs grass —
  hue-distant, high contrast
- dark grey walls `#5a5a6a`, `#6a6560` (render.js:
  526, 530) vs grass — neutral-on-green, fine
- sand/tavern `#a08850` (render.js:409) vs grass —
  warm-on-cool, readable

Don't see a "these two are the same color" bug. The
composition is well-separated. The season-blind sky
is the bigger palette issue.
